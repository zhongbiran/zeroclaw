Configuración de múltiples modelos y cadenas de respaldo

Un recorrido por los patrones comunes para usar múltiples proveedores de modelos: optimización de costos, niveles de calidad, enfoque local con respaldo en la nube, rotación de claves de API y resiliencia ante límites de tasa.

El material de referencia para el sistema del proveedor se encuentra en:

  • Proveedores de modelos → Descripción general — qué son los proveedores, forma de configuración
  • Proveedores de modelos → Fallback y enrutamiento — meta-proveedores reliable y router
  • Proveedores de modelos → Catálogo — la forma de configuración de cada proveedor

Cuándo usar una configuración de múltiples modelos

La configuración de múltiples modelos es útil para:

  • Alta fiabilidad: Cambia automáticamente a proveedores alternativos cuando el principal falla
  • Optimización de costos: Enruta modelos costosos a través de cadenas de respaldo para escenarios con límites de tasa
  • Resiliencia regional: Utiliza proveedores distribuidos geográficamente para gestionar las interrupciones específicas de cada región.
  • Flexibilidad de capacidades: Prueba diferentes modelos cuando uno carezca de las características necesarias (por ejemplo, llamada de herramientas, visión).
  • Manejo de límites de tasa: Rota las claves de API en las respuestas 429 (límite de tasa)
  • Desarrollo y pruebas: Cambia entre modelos en la nube y locales sin realizar cambios en el código

Conceptos principales

Cadenas de proveedores de respaldo

Cuando un proveedor experimenta un error transitorio (timeout, fallo de conexión, problema de autenticación), ZeroClaw intenta automáticamente los proveedores de respaldo en el orden especificado.

Ejemplo: Si tu proveedor principal es openai pero está temporalmente no disponible, ZeroClaw puede automáticamente hacer una transición a anthropic, luego a groq.

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]

Cuando el proveedor principal se recupera, ZeroClaw reanuda su uso (sin failover persistente).

Fallbacks a nivel de modelo

Algunos modelos pueden no estar disponibles en todas las regiones, o quizás quieras usar un modelo más rápido cuando un modelo pesado esté sujeto a límites de tasa.

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6", "gpt-4o"] }

Si claude-opus-4-7 falla o no está disponible, ZeroClaw intenta los modelos de respaldo en orden, manteniéndose dentro del mismo proveedor (a menos que también se haya configurado un respaldo a nivel de proveedor).

Rotación de claves API

Para los proveedores que frecuentemente encuentran límites de tasa, puedes proporcionar claves de API adicionales que ZeroClaw rotará en las respuestas 429.

[reliability]
api_keys = ["sk-key-2", "sk-key-3", "sk-key-4"]

La api_key principal (configurada globalmente o por canal) siempre se intenta primero; estos extras se rotan en caso de errores de límite de tasa.

Reintentos del proveedor

Cada intento del proveedor incluye reintentos configurables con retroceso exponencial antes de pasar al siguiente fallback.

[reliability]
provider_retries = 2          # Retry count per provider
provider_backoff_ms = 500     # Initial backoff in milliseconds

Estructura de configuración

Todo el comportamiento multimodelo se encuentra en la sección [reliability] de config.toml. Consulta la Referencia de configuración para obtener el índice completo de campos y los valores predeterminados.

Ejemplos de configuraciones

Cadena básica de respaldo

Configura un simple respaldo desde tu proveedor principal a uno de respaldo:

default_provider = "openai"
default_model = "gpt-4o"

[reliability]
fallback_providers = ["anthropic"]

Comportamiento: Si OpenAI supera el tiempo de espera o devuelve un error, ZeroClaw reintentará dos veces con retroceso exponencial y luego intentará la misma solicitud utilizando Anthropic.

Configuración de Alta Disponibilidad con Múltiples Proveedores

Combina las opciones de respaldo del proveedor con las opciones de respaldo del modelo y la rotación de claves de API:

default_provider = "openai"
default_model = "gpt-4o"
api_key = "sk-openai-primary"

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]
api_keys = ["sk-openai-backup-1", "sk-openai-backup-2"]

[reliability.model_fallbacks]
"gpt-4o" = ["gpt-4-turbo", "gpt-3.5-turbo"]
"gpt-4-turbo" = ["gpt-3.5-turbo"]

Comportamiento:

  1. Prueba OpenAI gpt-4o con la clave principal (2 reintentos)
  2. En caso de límite de tasa, cambiar a claves de API de respaldo
  3. Si OpenAI sigue fallando, vuelve a Anthropic con la misma solicitud de modelo (Anthropic seleccionará el equivalente disponible).
  4. Si Anthropic no está disponible, prueba con Groq y luego con OpenRouter.
  5. Si el modelo no está disponible, intenta usar los modelos de respaldo en orden

Desarrollo local con respaldo en la nube

Utiliza una instancia local de Ollama como principal y, en caso de fallo, recurre a un proveedor en la nube:

default_provider = "ollama"
default_model = "llama2:70b"
api_url = "http://localhost:11434"

[reliability]
fallback_providers = ["openrouter", "groq"]

Comportamiento: Si Ollama se cae o supera el tiempo de espera, utiliza automáticamente OpenRouter o Groq en su lugar sin necesidad de cambios en la configuración.

Optimización de costos: Modelo pesado con fallback rápido

Utiliza un modelo de razonamiento costoso para tareas complejas, pero recurre a un modelo más rápido:

default_provider = "anthropic"
default_model = "claude-opus-4-7"

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6"] }

Comportamiento: Cuando Opus está limitado por la velocidad o es lento, utiliza automáticamente Sonnet (generalmente 2–3 veces más rápido y económico).

Configuración de múltiples regiones

Para organizaciones con implementaciones en múltiples regiones:

# Región principal de EE. UU.
default_provider = "anthropic"
default_model = "claude-sonnet-4-6"

[reliability]
# Recurrir al proveedor de la región de la UE si Anthropic en EE. UU. no está disponible
fallback_providers = ["bedrock"]  # AWS Bedrock en múltiples regiones
provider_retries = 3
provider_backoff_ms = 1000

Asegúrate de que cada proveedor de respaldo tenga credenciales en tu entorno:

export ANTHROPIC_API_KEY="..."
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."

Comportamiento de Recarga Rápida

La sección [reliability] es recargable en caliente. Mientras un canal o puerta de enlace está en ejecución, las actualizaciones en config.toml surten efecto en el siguiente mensaje entrante sin necesidad de reiniciar.

Manejo de errores y activadores de respaldo

El fallback se activa por:

  • Tiempo de espera agotado: El proveedor no respondió dentro del tiempo de espera configurado
  • Error de conexión: Fallo de red/DNS
  • Error de autenticación: Credenciales no válidas (solo se reintenta si se detectan problemas transitorios del servicio de autenticación)
  • Límite de tasa (429): HTTP 429; activa primero la rotación de la clave de API y luego el cambio de proveedor
  • Servicio no disponible (503): Problema temporal del servicio
  • Modelo no encontrado: Activa la cadena de respaldo del modelo si está configurada

El fallback no se activa por:

  • Solicitud no válida (400): Entrada malformada; reintentar no ayudará
  • Error de autenticación permanente: Formato de clave API no válido
  • Errores de salida del modelo: El modelo respondió pero devolvió un error

Depuración de la actividad de respaldo

Habilitar trazas en tiempo de ejecución para depurar el comportamiento de respaldo:

[observabilidad]
modo_rastro_ejecucion = "rodante"
ruta_rastro_ejecucion = "state/runtime-trace.jsonl"

Luego, consulta los rastros:

# Mostrar todos los eventos de respaldo
zeroclaw doctor traces --contains "fallback"

# Mostrar detalles de reintento del proveedor
zeroclaw doctor traces --contains "proveedor"

# Mostrar rotación de límite de tasa
zeroclaw doctor traces --contains "429"

Mejores prácticas

  1. Ordenar por fiabilidad: Coloca los proveedores más confiables primero en fallback_providers.
  2. Probar cadenas de respaldo: Verificar el comportamiento de respaldo antes de su uso en producción
  3. Monitorizar la rotación de la clave API: Rastrear eventos de límite de tasa para saber cuándo está activa la rotación
  4. Mantén las alternativas del modelo semánticamente similares: No cambies de un modelo de razonamiento a un modelo de chat sin intención
  5. Usa variables de entorno: Almacena las claves API sensibles en variables de entorno, no en archivos de configuración.
  6. Intención de respaldo del documento: Agrega comentarios en la configuración explicando por qué existe cada respaldo.
  7. Verificar las credenciales de múltiples modelos: Asegúrate de que todos los proveedores de respaldo tengan credenciales válidas configuradas

Resolución de credenciales

Cada proveedor de respaldo resuelve las credenciales de forma independiente utilizando el orden de resolución estándar:

  1. Credenciales explícitas desde la configuración/CLI
  2. Variable de entorno específica del proveedor
  3. Valor predeterminado genérico: ZEROCLAW_API_KEY, luego API_KEY

Importante: La clave API del proveedor principal no se reutiliza automáticamente por los proveedores de respaldo. Establezca las credenciales para cada proveedor por separado.

Ejemplo:

export OPENAI_API_KEY="sk-..."
export ANTHROPIC_API_KEY="claude-..."
export GROQ_API_KEY="gsk-..."

Límites y restricciones

  • Proveedores de respaldo máximo: Limitado por el tamaño del archivo de configuración (generalmente se admiten más de 100 cadenas)
  • Caídas máximas del modelo por modelo: Sin límite estricto
  • Rotación de claves API: Se prueban todas las claves antes de agotar el tiempo de espera
  • Intentos de reintento: Configurables por proveedor con retroceso exponencial
  • Presupuesto total de tiempo de espera: Acumulado entre reintentos y alternativas; el tiempo de espera a nivel de canal sigue siendo aplicable
