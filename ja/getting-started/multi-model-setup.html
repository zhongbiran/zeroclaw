マルチモデル設定とフォールバックチェーン

複数のモデルプロバイダーを使用するための一般的なパターンを解説します。コスト最適化、品質ティア、ローカルファーストでホストされたフォールバック、APIキーのローテーション、レートリミットの耐性などについて説明します。

プロバイダーシステムのリファレンス資料は以下にあります：

  • モデルプロバイダー → 概要 — プロバイダーとは何か、設定の構造
  • モデルプロバイダー → フォールバックとルーティング — reliable および router メタプロバイダー
  • モデルプロバイダー → カタログ — 各プロバイダーの構成形状

マルチモデル設定を使用する場合

マルチモデル設定は以下の場合に役立ちます：

  • 高い信頼性: プライマリプロバイダーが失敗した場合、自動的に代替プロバイダーにフォールバック
  • コスト最適化: レート制限シナリオで高コストモデルをフォールバックチェーン経由でルーティング
  • 地域的耐障害性: 地理的に分散したプロバイダーを使用して地域固有の障害に対応
  • 機能の柔軟性: 必要な機能がないモデル（ツール呼び出しやビジョンなど）がある場合、異なるモデルを試す
  • レート制限処理: 429（レート制限）レスポンスでAPIキーをローテーション
  • 開発とテスト: コードを変更せずにクラウドモデルとローカルモデルを切り替え

フォールバックプロバイダーチェーン

プロバイダーが一時的なエラー（タイムアウト、接続失敗、認証エラー）に遭遇すると、ZeroClawは指定された順序でフォールバックプロバイダーを自動的に試みます。

例: プライマリプロバイダーが openai だが一時的に利用不可の場合、ZeroClawは自動的に anthropic にフォールバックし、次に groq を試します。

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]

プライマリプロバイダーが復旧すると、ZeroClawはそれを使用再開します（スティッキーフェイルオーバーなし）。

モデルレベルのフォールバック

一部のモデルはすべてのリージョンで利用できない場合があります。または、重いモデルがレート制限されている場合により高速なモデルを使用したい場合があります。

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6", "gpt-4o"] }

claude-opus-4-7 が失敗または利用不可の場合、ZeroClawはプロバイダーレベルのフォールバックも設定されていない限り、同じプロバイダー内でフォールバックモデルを順番に試します。

APIキーローテーション

頻繁にレート制限に遭遇するプロバイダーの場合、ZeroClawが 429 レスポンスでローテーションするための追加APIキーを指定できます。

[reliability]
api_keys = ["sk-key-2", "sk-key-3", "sk-key-4"]

プライマリ api_key（グローバルまたはチャネルごとに設定）は常に最初に試され、これらの追加キーはレート制限エラーでローテーションされます。

プロバイダー再試行

各プロバイダーの試行は、次のフォールバックに移動する前に、指数バックオフを使用した再試行可能な再試行を含みます。

[reliability]
provider_retries = 2          # Retry count per provider
provider_backoff_ms = 500     # Initial backoff in milliseconds

設定構造

すべてのマルチモデル動作は config.toml の [reliability] セクションに配置されます。全フィールドインデックスとデフォルトについてはConfig リファレンスを参照してください。

設定例

基本的なフォールバックチェーン

プライマリプロバイダーからバックアップへの単純なフォールバックを設定します：

default_provider = "openai"
default_model = "gpt-4o"

[reliability]
fallback_providers = ["anthropic"]

動作: OpenAIがタイムアウトまたはエラーを返した場合、ZeroClawは指数バックオフで2回再試行し、その後Anthropicを使用して同じリクエストを試みます。

高信頼性マルチプロバイダー設定

プロバイダーフォールバックをモデルフォールバックおよびAPIキーローテーションと組み合わせます：

default_provider = "openai"
default_model = "gpt-4o"
api_key = "sk-openai-primary"

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]
api_keys = ["sk-openai-backup-1", "sk-openai-backup-2"]

[reliability.model_fallbacks]
"gpt-4o" = ["gpt-4-turbo", "gpt-3.5-turbo"]
"gpt-4-turbo" = ["gpt-3.5-turbo"]

動作：

  1. プライマリキーで OpenAI gpt-4o を試行（2回の再試行）
  2. レート制限時にバックアップAPIキーにローテーション
  3. OpenAIが依然として失敗した場合、同じモデルリクエストでAnthropicにフォールバック（Anthropicが利用可能な同等のものを選択）
  4. Anthropropicが利用不可の場合、Groq、次にOpenRouterを試す
  5. モデルが利用不可の場合、フォールバックモデルを順番に試す

クラウドフォールバック付きローカル開発

ローカルOllamaインスタンスをプライマリとして使用し、クラウドプロバイダーにフォールバック：

default_provider = "ollama"
default_model = "llama2:70b"
api_url = "http://localhost:11434"

[reliability]
fallback_providers = ["openrouter", "groq"]

動作: Ollamaがダウンするか、タイムアウトする場合、設定の変更なしに自動的にOpenRouterまたはGroqを使用します。

コスト最適化：高コストモデルと高速フォールバック

複雑なタスク用に高コストの推論モデルを使用しますが、より高速なモデルにフォールバック：

default_provider = "anthropic"
default_model = "claude-opus-4-7"

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6"] }

動作: Opusがレート制限されるか遅い場合、自動的にSonnet（通常2～3倍高速かつ安い）を使用します。

マルチリージョンセットアップ

マルチリージョンデプロイメントを行う組織の場合:

# Primary US region
default_provider = "anthropic"
default_model = "claude-sonnet-4-6"

[reliability]
# Fall back to EU region provider if US Anthropic is down
fallback_providers = ["bedrock"]  # AWS Bedrock in multiple regions
provider_retries = 3
provider_backoff_ms = 1000

各フォールバックプロバイダーが環境に認証情報を持っていることを確認してください:

export ANTHROPIC_API_KEY="..."
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."

ホットリロード動作

[reliability] セクションはホットリロード可能です。チャネルまたはゲートウェイが実行中の場合、config.toml への更新は再起動なしで次のインバウンドメッセージで有効になります。

エラーハンドリングとフォールバックトリガー

フォールバックがトリガーされる条件:

  • タイムアウト: プロバイダーが構成されたタイムアウト内に応答しなかった
  • 接続エラー: ネットワーク/DNS障害
  • 認証エラー: 無効な認証情報(一時的な認証サービス問題が検出された場合のみ再試行)
  • レート制限 (429): HTTP 429; まずAPIキーローテーションをトリガーし、次にプロバイダーフォールバック
  • サービス利用不可 (503): 一時的なサービス問題
  • モデルが見つからない: 構成されている場合、モデルフォールバックチェーンをトリガー

フォールバックがトリガーされない条件:

  • 無効なリクエスト (400): 不正な入力; 再試行しても効果なし
  • 永続的な認証失敗: 無効なAPIキー形式
  • モデル出力エラー: モデルが応答したがエラーを返した

フォールバック活動のデバッグ

ランタイムトレースを有効にしてフォールバック動作をデバッグします:

[observability]
runtime_trace_mode = "rolling"
runtime_trace_path = "state/runtime-trace.jsonl"

次にトレースをクエリします:

# すべてのフォールバックイベントを表示
zeroclaw doctor traces --contains "fallback"

# プロバイダー再試行の詳細を表示
zeroclaw doctor traces --contains "provider"

# レート制限ローテーションを表示
zeroclaw doctor traces --contains "429"

ベストプラクティス

  1. 信頼性で順序付け: fallback_providers に最も信頼性の高いプロバイダーを最初に配置
  2. フォールバックチェーンをテスト: 本番環境での使用前にフォールバック動作を確認
  3. APIキーローテーションを監視: ローテーションが活動中の時期を把握するためにレート制限イベントを追跡
  4. モデルフォールバックを意味的に同様に保つ: 推論モデルからチャットモデルへの意図しないフォールバックは避ける
  5. 環境変数を使用: 機密APIキーは設定ファイルではなく環境に保存
  6. フォールバック意図を文書化: 各フォールバックが存在する理由を説明するコメントを設定に追加
  7. マルチモデル認証情報を確認: すべてのフォールバックプロバイダーが有効な認証情報セットを持っていることを確認

認証情報の解決

各フォールバックプロバイダーは標準解決順序を使用して認証情報を独立して解決します:

  1. 設定/CLIからの明示的な認証情報
  2. プロバイダー固有の環境変数
  3. 汎用フォールバック: ZEROCLAW_API_KEY、次に API_KEY

重要: プライマリプロバイダーのAPIキーは、フォールバックプロバイダーによって自動的に再利用されません。各プロバイダーの認証情報を個別に設定してください。

例:

export OPENAI_API_KEY="sk-..."
export ANTHROPIC_API_KEY="claude-..."
export GROQ_API_KEY="gsk-..."

制限と制約

  • 最大フォールバックプロバイダー数: 設定ファイルサイズによって制限(通常100以上のチェーンがサポート)
  • モデルごとの最大モデルフォールバック数: ハードリミットなし
  • APIキーローテーション: すべてのキーがタイムアウト前に試行されます
  • 再試行回数: プロバイダーごとに設定可能、指数バックオフ付き
  • 総タイムアウト予算: 再試行とフォールバック全体で累積; チャネルレベルのタイムアウトは引き続き適用されます
