ドキュメントと翻訳

ZeroClawには2つの独立した翻訳レイヤーがあります：

レイヤーフォーマットカバーする内容
アプリの文字列Mozilla Fluent (.ftl)CLIヘルプテキスト、コマンドの説明、ランタイムメッセージ
ドキュメントgettext (.po)このmdBookのすべての内容

これらは個別に埋め込まれ、個別に保存されます。どちらもプロバイダーに依存しない埋め込みパイプラインを使用しており、~/.zeroclaw/config.toml の [providers.models.<name>] セクションで OpenAI 互換のエンドポイントを構成し、埋め込みコマンドに --provider <name> を渡すことで利用できます。

Ollama を介したローカルモデルは第一級のオプションです — APIキーは不要で、呼び出しごとのコストもかかりません。リリース品質にはホスト型プロバイダーも適しています。翻訳はローカルで行うため、PR を作成する前に cargo mdbook sync を実行してください。

プロバイダー設定

Ollama は現在、ドキュメントの公式ソースです。Ollama がインストールされ、qwen3.6:35-a3b がプルされていることを確認してください。その後、~/.zeroclaw/config.toml（または設定ファイルの保存先）で:

# Ollama経由のローカル設定 — 無料で、マシン上で動作
[providers.models.ollama]
base_url = "http://localhost:11434"
model = "qwen3.6:35b-a3b" # 現在推奨のモデル

ドキュメントをローカルでビルドする

ドキュメントをローカルでビルドする

いまご覧のドキュメントサイトは docs/book/ から公開されています。同じサイトをご自身のマシンでビルドすることもできます — オフラインで読みたいとき、PR を出す前に編集内容をプレビューしたいとき、翻訳を編集したいときに便利です。

ワンコマンドで実行

cargo mdbook serve                       # http://localhost:3000/en/ ですべてのロケールを配信
cargo mdbook serve --locale ja           # 日本語ソースに対するライブリロード
cargo mdbook build                       # すべてのロケールの静的ビルドを docs/book/book/ に
cargo mdbook refs                        # 自動生成された参照ページを再生成する
cargo mdbook sync                        # 英語ソースを編集した後: 再抽出 + AI で差分を埋める
cargo mdbook sync --locale ja            # 1 つのロケールのみ同期する
cargo mdbook sync --force                # すべてを強制的に再翻訳する (品質チェック)
cargo mdbook sync --locale ja --force    # 1 つのロケールのみ強制的に再翻訳する
cargo mdbook stats                       # ロケールごとに翻訳済み/ファジー/未翻訳を表示する
cargo mdbook check                       # .po フォーマットを検証する (翻訳 PR の前に実行)

必要なツール

cargo mdbook は、不足しているものを即座に報告して失敗しますが、参考までに：

何がどこにビルドされるか

ソース出力生成者
docs/book/src/**/*.md (手書き)docs/book/book/<locale>/mdbook build
docs/book/src/reference/cli.md(同じパス; gitignored)cargo run -- markdown-help
docs/book/src/reference/config.md(同じパス; gitignored)cargo run -- markdown-schema

2つの reference/*.md ファイルは、実際のコード内の clap 派生物とJSONスキーマから生成されます — 決して手で編集しないでください。代わりに、関連するRust型の /// ドキュメンテーションコメントを編集してください。

翻訳を最新に保つ方法

英語マークダウンは人間によって保守されている唯一のソースです。翻訳は docs/book/po/<locale>.po ファイルに保存され、これはキャッシュとして機能します — ドキュメントのコピーではありません。

英語のソースが変更されると、cargo mdbook sync は2つのステージを実行します。

  1. 抽出: mdbook-xgettext は現在の英語ソースから po/messages.pot を再生成します
  2. マージ: msgmerge は各ロケールの .po ファイルを更新します — 新しい文字列は空の msgstr "" を取得します; 変更された文字列は #, fuzzy でマークされ、古い翻訳は出発点として保持されます

その後、このコマンドはfuzzy + 未翻訳のエントリをカウントします。差分があり、--provider が指定されている場合、fill-translations ツールはこれらのエントリのみを翻訳します。変更のない文字列のコストはゼロです。.po ファイルのキャッシュにより、変更のないソースに対して再実行しても何もしません。

--provider を指定しない場合、cargo mdbook sync は依然として extract + merge を実行し、翻訳が必要な文字列の数を報告します。msgstr がない文字列はレンダリング時に英語にフォールバックします。部分的な翻訳も有効です。

新しいロケールの追加

  1. リポジトリのルートにある locales.toml を編集してください — これだけのファイルを変更すればOKです：

    [[locale]]
code = "xx"
label = "言語名"

    それ以外のもの（lang-switcher.js、CI、cargo fluent fill、cargo mdbook sync）はこのファイルから自動的に読み取ります。

  2. ドキュメントの翻訳を初期化して埋める:

    cargo mdbook sync --locale xx --provider ollama

    .po ファイルが存在しない場合は自動的にブートストラップされ、すべてのエントリが埋められます。

  3. 検証とプレビュー:

    cargo mdbook check              # フォーマットエラーで0以外で終了します
cargo mdbook stats              # カバレッジカウントを表示します
cargo mdbook serve --locale xx

リリース翻訳ワークフロー

リリースをタグ付けする前に、ローカルで完全な翻訳パスを実行し、更新された .po ファイルをコミットしてください。

# ファストデルタパス（前回のリリース以降に追加または変更された文字列のみ）
cargo mdbook sync --provider ollama

# または：品質チェック — すべてを再翻訳
cargo mdbook sync --provider ollama --force

cargo mdbook check   # コミット前に検証します
cargo mdbook stats   # カバレッジを確認します

使用されるモデルは、config.toml の [providers.models.<name>] で設定されているものになります。

ヒント

  • プロスの高速反復: cargo mdbook serve は保存時に自動で再構築します。CLI フラグや構成スキーマを変更した場合を除き、cargo mdbook refs をスキップしてください。
  • 翻訳の高速イテレーション: po/<locale>.po を編集してブラウザをリロードしてください — mdbook serve は .po の変更を検出して自動的に再構築されます。
  • クリーンアップ: rm -rf docs/book/book target/doc は生成されたすべてのものを削除します。
  • ゼロコストの再実行: 変更のない英語のソースに対して cargo mdbook sync を実行すると、数秒で完了します — AI呼び出しなし、コストなし。

アプリの文字列を埋め込む（Fluent）

アプリの文字列は crates/zeroclaw-runtime/locales/ にあります。英語がソースオブトゥルースであり、コンパイル時に埋め込まれます。非英語のロケールは、実行時に ~/.zeroclaw/workspace/locales/ から読み込まれます。

cargo fluent stats                                          # ロケールごとのカバレッジ
cargo fluent check                                          # .ftl 構文を検証する
cargo fluent fill --locale ja --provider ollama             # 欠落しているキーを埋める（デフォルトのバッチサイズは50）
cargo fluent fill --locale ja --provider ollama --batch 1   # 1ファイルずつ処理（バッチ50で切り捨てられる長いエントリを含むファイルの場合に使用、例：tools.ftl）
cargo fluent fill --locale ja --provider ollama --force     # すべてを再翻訳
cargo fluent scan                                           # Rust ソースコードとの間で、期限切れまたは欠落したキーを検出する

各バッチは次のAPI呼び出しの前にディスクに書き込まれるため、実行中の失敗は現在処理中のバッチのみを失うだけです。再実行では、既にターゲットの .ftl に存在するキーはスキップされるため、再開は自動的に行われ、--force は不要です。

更新された .ftl ファイルをワークスペースにコピーし、変更を反映させるために再ビルドしてください：

mkdir -p ~/.zeroclaw/workspace/locales/ja
cp crates/zeroclaw-runtime/locales/ja/cli.ftl ~/.zeroclaw/workspace/locales/ja/cli.ftl

ドキュメントの翻訳（gettext）

ドキュメントの翻訳は docs/book/po/ に格納されます。cargo mdbook sync は、抽出 → 結合 → 不要な文字列の削除 → AIによる翻訳補完を一度に実行します。--provider を指定しない場合、sync は抽出と結合を実行し、翻訳が必要な文字列の数を報告します。部分的に翻訳された文字列は、レンダリング時に英語にフォールバックします。

cargo mdbook sync --provider ollama              # デルタフィル
cargo mdbook sync --provider ollama --force      # 品質パス：すべてのエントリを再翻訳
cargo mdbook sync --provider ollama --batch 1    # 逐次処理（不安定なローカルモデルに有用）
cargo mdbook sync --locale ja --provider ollama  # 単一ロケール

パイプラインには組み込みの耐障害性があります：

  • リーク検出 — モデルが翻訳の代わりに自身の指示を返した場合、ツールはパターンを検出（応答の長さの比率と箇条書き構造により）、応答の末尾から実際の翻訳を復元しようと試み、復元が失敗した場合は再翻訳のために該当エントリを空白にします。
  • インクリメンタルな書き込み — 各バッチの後に .po ファイルが再書き込みされます。実行中に Ctrl-C を押しても、その時点までの進捗は失われません。
  • 非推奨のストリッピング — msgmerge + msgattrib --no-obsolete は、削除されたソース文字列が #~ エントリとして蓄積するのを防ぎます。

新しいロケールの追加

  1. リポジトリのルートにある locales.toml を編集してください — これだけのファイルを変更すればOKです：

    [[locale]]
code = "<code>"
label = "言語名"

  2. アプリの文字列を翻訳してください。

    cargo fluent fill --locale <code> --provider ollama

  3. ドキュメントの .po ファイルを初期化して埋める:

    cargo mdbook sync --locale <code> --provider ollama

それ以外のもの（lang-switcher.js、CI のデプロイ対象リスト、cargo mdbook locales の出力）は、locales.toml から自動的に読み取られます。

モデル品質に関する注記

言語やモデルによって翻訳品質は大きく異なります。

ロケールよくサポートされている注釈
ja, zh-CNQwen3.6ファミリー、または他の最先端のホスト型モデルについてお探しでしょうか？もし具体的な情報や比較が必要な場合は、詳細をお知らせください。Qwenは中国語を第一言語としていますが、日本語も強力です。
es, frqwen3.6、mistral、gemma3、ホストロマンス語は広範にわたってよく訓練されている
低リソースのロケールホストされたフロンティアモデルのみローカルモデルはしばしば単語を誤って生成することがあります。

リリース品質のパスには、--force オプションを使用してホストされたフロンティアモデルを優先してください。開発中の継続的なデルタフィリングには、ローカルの Ollama モデルが適しており、無料で利用できます。
