安全性 — 概述

能够执行 shell 命令、打开 URL 并写入文件的代理是一个特权进程。ZeroClaw 的安全模型作用于每次工具调用和每条通道消息之上，在运行时限制代理实际允许执行的操作。

共有六层，从外到内依次为：

1. 通道配对与访问控制

在频道消息到达代理之前，会检查频道的配对和允许列表。allowed_users、allowed_chats、Webhook 的 IP 允许列表——所有这些都在频道适配器层面强制执行，在运行时看到事件之前。

文档：每个频道页面位于 频道 下。

2. 自主性级别

粗粒度旋钮。三个设置：

  • ReadOnly — 智能体可以观察（读取文件、查询记忆、获取其被允许获取的 URL），但不能写入或执行命令。
  • 监督式（默认）——低风险操作运行；中风险操作询问操作员；高风险操作阻止。
  • Full — no approval gates; workspace_only is implicitly disabled. forbidden_paths, forbidden_commands, and the OS sandbox still enforce.

文档：自主级别。

3. 工作区边界和路径规则

代理在配置的工作区目录中运行。除非设置 workspace_only = false，否则 file_read、file_write 和 shell（用于操作文件系统的命令）将拒绝访问该目录之外的路径。

Per-session sandbox roots (ACP and gateway WebSocket): When a session is opened via ACP (session/new with a cwd parameter) or via the gateway WebSocket (connect-time cwd parameter), that path becomes the SecurityPolicy workspace boundary for all file and shell tools for the lifetime of the session. The daemon’s global workspace_dir remains the data directory for memory, identity, cron, and other persistent state. The model is: session cwd = project boundary the agent can touch; workspace_dir = where ZeroClaw stores its own files. Note: the agent’s system prompt currently reflects the daemon’s workspace_dir rather than the session cwd; enforcement is correct but the model’s self-reported location may differ.

Important: the cwd parameter changes which directory on the ZeroClaw host the agent is sandboxed to — it does not affect which machine tools run on. Tool use (shell commands, file reads/writes) always executes on the machine running ZeroClaw. If you connect to a remote ZeroClaw instance over the gateway WebSocket, tool calls operate on the remote machine’s filesystem, not on your local machine. For localhost-only deployments this distinction does not matter, but remote setups should account for it.

在工作区之外，forbidden_paths 列表（默认值：/etc、/sys、/boot、~/.ssh 等）始终会被阻止，不受工作区设置的影响。

4. Shell 命令策略

对于 shell 调用：

  • allowed_commands — 如果非空，则 shell 仅运行其 basename 在此列表中的命令
  • forbidden_commands — 显式禁止列表（rm -rf /、shutdown、内核操作）
  • validate_command_execution — 一个模式匹配传递，用于查找危险的标志、管道和参数形状

验证器在命令执行到 shell 之前运行。被阻止的命令会作为工具错误呈现给模型，模型可以对此做出反应。

5. 操作系统级沙箱

当有沙箱后端可用时，工具调用将在其中运行：

平台默认后端
LinuxLandlock（内核）/ Bubblewrap / Firejail / Docker — 自动检测
macOS安全带（原生）
WindowsAppContainer（实验性）
任何Docker（如果守护进程可访问）

沙箱将文件系统访问限制在工作区范围内，仅保留工具明确需要的网络可达性，并移除对父进程密钥的访问权限。

文档：沙箱化。

6. 工具收据

每次工具调用——无论其是否执行、被阻止或需要批准——都会在链中生成一个签名收据。每个收据都包含前一个收据的哈希值，因此篡改任何收据都会使其余收据失效。

收据是“代理昨天做了什么”的真实来源。它们可读、可搜索且持久。

文档：工具收据。

附加门

超越这六层：

  • OTP 门控 — [security.otp] gated_actions = ["shell", "browser", "file_write"] 要求在列出每个操作之前提供一次性代码。适用于远程访问场景。
  • 紧急停止 — zeroclaw estop 会中止所有正在进行的工具调用。当 [security.estop] enabled = true 时，恢复操作需要一次性密码（OTP）。
  • 提示注入防护 — 在验证工具调用之前，扫描模型输出以检测已知的注入模式。
  • 泄漏检测器 — 扫描出站消息中的敏感信息（如 API 密钥模式、私钥），并阻止发送匹配的内容。
  • 配对保护 — 用于通道身份验证的设备配对；防止被盗用的凭据在新设备上生效。

当出现问题时

被阻止的工具调用不会静默失败：

  1. 安全验证器返回错误
  2. 运行时将其包装为 ToolResult::Err 并返回给模型
  3. 模型看到“错误：Shell 命令被策略阻止：禁止的模式 rm -rf /”，可以重试、道歉或询问用户

如果某个频道位于受限工具集（tools_allow = [...]）中，则该工具不会向该频道的模型展示。模型永远不会看到它无法使用的工具。

默认姿势

开箱即用：

  • 自主性：监督
  • 仅工作区：true
  • 沙盒：自动检测（使用操作系统提供的功能）
  • 审计日志记录：false（显式启用）
  • OTP：false
  • 急停：false

这是一个合理的折中方案——对笔记本电脑来说足够安全，同时也不会让人感到沮丧。在生产环境中可以调高（OTP、审计、受限工具），或者在开发机上调低到 YOLO 模式。
