替代 PR

当维护者提交的 PR 替换了贡献者的开放 PR 时，归属关系和流程纪律有助于保持贡献者关系的健康。本页面是规则手册。

先尝试其他替代方案

Superseding 是最重的选项。在打开它之前，请按以下顺序尝试：

  1. 将修复提交推送到贡献者的分支。 如果 PR 具有 maintainerCanModify: true（来自个人 fork 的 PR 的默认值 — 可通过 gh pr view <number> --json maintainerCanModify 确认），则直接推送你的修复提交并合并贡献者的 PR。git log、git blame 以及贡献者的 GitHub 个人资料中的署名将保持清晰。如果你的修复并非微不足道，请先与贡献者协调 — 在他们有未推送的更改时推送会导致他们需要解决的冲突。

  2. 提出包含具体所需更改的审查意见。 如果贡献者响应积极，且修复内容在其原始范围内（例如 clippy 警告、边界情况、测试用例添加），请提出更改请求，并让他们推送修复提交。单行修复几乎总是比覆盖提交更适合作为更改请求。

  3. 合并后提交一个后续 PR。 如果贡献者的 PR 本身是正确的，而你希望进一步加固代码，请先合并该 PR，然后再提交一个单独的 PR。贡献者署名得以保留；代价是 master 分支上会短暂存在已知问题。

仅在以下情况之一适用时，才进行替代：

  • 贡献者未响应（在项目审查 SLA 内无回复）。
  • 该变更需要的工作量远超贡献者最初的范围。
  • 多个相关的贡献者 PR 需要合并为一个连贯的变更。
  • 贡献者选择不接受维护者编辑（maintainerCanModify: false），且后续 PR 不切实际。

归属规则

当你进行替代（supersede）并延续实质性的代码或设计决策时，请明确保留作者信息：

  • 为每个其工作被实质性合并的已替代贡献者添加一个 Co-authored-by: Name <email> 尾注。使用 GitHub 认可的邮箱——可以是贡献者的 <login@users.noreply.github.com> 格式，也可以是他们的已验证提交邮箱。
  • trailers 应位于提交消息末尾的空白行之后的单独行上。切勿将其编码为转义的 \n 文本。
  • 在 PR 正文中，列出被取代的 PR 链接，并简要说明从每个 PR 中合并了哪些内容。
  • 如果没有实际代码或设计被纳入（仅作为灵感），请勿使用 Co-authored-by，而是在 PR 备注部分给予致谢。

这些 trailer 能正确路由 GitHub 的贡献者识别信息。如果没有它们，原始作者在其 PR 上会显示为“已关闭”，且没有保留后续贡献的记录。

PR 标题和正文模板

feat(<scope>): unify and supersede #<pr_a>, #<pr_b> [and #<pr_n>]

## Supersedes

- #<pr_a> by @<author_a>
- #<pr_b> by @<author_b>

## Integrated scope

- From #<pr_a>: <what was materially incorporated>
- From #<pr_b>: <what was materially incorporated>

## Attribution

- `Co-authored-by` trailers added for materially incorporated contributors: Yes/No
- If No, explain why

## Non-goals

- <explicitly list what was not carried over>

## Risk and rollback

- Risk: <summary>
- Rollback: <revert commit/PR strategy>

提交消息模板

feat(<scope>): unify and supersede #<pr_a>, #<pr_b> [and #<pr_n>]

<one-paragraph summary of integrated outcome>

Supersedes:
- #<pr_a> by @<author_a>
- #<pr_b> by @<author_b>

Integrated scope:
- <subsystem_or_feature_a>: from #<pr_x>
- <subsystem_or_feature_b>: from #<pr_y>

Co-authored-by: <Name A> <login_a@users.noreply.github.com>
Co-authored-by: <Name B> <login_b@users.noreply.github.com>

关闭已废弃的 PR

在每个末尾添加注释，注明新的 PR 和延续内容：

Superseded by #<new_pr>. Your work is incorporated as `Co-authored-by` —
specifically the <X> approach in <Y>. Thanks for the original take here;
closing this one in favor of the unified PR.

如果贡献者在原始 PR 中对某个设计选择提出了异议，而后续版本采取了不同的方向，请明确指出这一点。不要将其伪装成干净的延续，而实际上却是重新设计。

交接模板（代理 → 代理 或 代理 → 维护者）

在进行中途工作交接时，请包括以下内容：

  1. 变更内容。
  2. 未更改的内容。
  3. 验证运行及结果。
  4. 剩余风险/未知项。
  5. 下一步推荐操作。

这适用于跨越多个工作会话的替代、维护者之间由代理协助的交接，以及任何需要一人接手另一人正在进行的分支的情况。
