ZeroClaw Docs

审查者手册

用于审查 PR 和分类问题的操作模型。其规模设计旨在在高负载下保持高质量的审查；通过风险进行路由，使高风险路径获得所需的关注，而无需让每个小改动都经过相同的关卡。

有关实际的拉取请求获取流程和审查结论机制，请参阅 PR 审查协议。本页面是 操作模型；协议是 流程。

Labels are maintainer metadata. If the correct label is obvious and you have permission, fix it yourself before finalizing the review. Ask the author only when the right label choice is ambiguous or nobody with label permissions is available.

标准工作流程

五分钟摄入量

对于每一个新的 PR，在查看任何代码之前：

  1. 确认 PR 模板填写完整：摘要、验证证据、安全与隐私、兼容性、回滚方案（适用于中/高风险变更）。
  2. 确认标签存在且合理 —— size:*、risk:*、范围标签，以及适用时的贡献者层级。
  3. 确认 CI Required Gate 信号状态。
  4. 确认范围是一个重要的考量因素。除非对混合功能有明确的理由，否则包含多种功能的超大 PR 会被退回要求拆分。
  5. 确认隐私/数据卫生规则。请参阅 隐私 获取完整规则。

如果任何准入检查失败，请留下一条可操作的清单评论并停止。不要对未通过准入的 PR 进行深度审查——在这一层进行来回沟通的成本低于在代码差异已被分析之后。

快速通道清单（每个 PR）

  • 作用域边界明确且可信。
  • 验证命令已存在，且结果一致。
  • 用户可见的行为变更已记录。
  • 作者展示了对行为和影响范围的理解（尤其是对于 AI 辅助的 PR）。
  • 回滚路径是具体的——“revert”并不具体。
  • 兼容性和迁移影响是明确的。
  • 没有个人数据或敏感数据泄露到差异制品中；测试使用中性、项目范围内的占位符。
  • 命名和架构边界遵循项目约定（AGENTS.md、扩展示例）。

深度审查清单（仅限高风险项）

对于 risk: high 的 PR，请验证每个类别中的具体示例。一个具体实例胜过五个笼统的声明。

  • 安全边界：保留默认拒绝行为，避免意外扩大作用域。
  • 故障模式：错误处理明确，安全降级。
  • 契约稳定性：保留 CLI、配置或 API 兼容性，或提供迁移文档。
  • 可观测性：在不泄露敏感信息的情况下诊断故障。
  • 回滚安全性：回滚路径和影响范围清晰明确。

注释形状

优先使用清单式注释，每条注释明确一个结果：

  • 准备合并（说明原因）。
  • 需要作者操作（按顺序排列的阻塞项列表）。
  • 需要更深入的安全或运行时审查（说明具体风险及所需证据）。

模糊的注释会导致不必要的往返沟通。如果你发现自己正在写“这可能是一个问题”，请多花 30 秒，将其转化为具体的场景，或者直接删除该注释。

问题分类

相同的风险路由原则同样适用于问题（issues），但标签和信号有所不同。

分类标签

标签何时使用
r:需要复现缺少确定性复现步骤的错误报告。阻止对此进行更深入的分类处理。
r:support使用或帮助问题更适合在错误跟踪系统之外处理。
重复 / 无效非可操作噪声。以礼貌的提示关闭。
no-stale已接受的工作正在等待外部阻塞项。这样可以避免问题被自动标记为陈旧。

如果报告中的日志或有效载荷包含个人标识符或敏感数据，请在进行更深入的分诊之前请求脱敏。分诊过程不得传播此类暴露。

PR 积压清理

当审查需求超过容量时：

  1. 将活跃的 bug 和安全 PR（size: XS/S）保持在队列顶部。
  2. 请要求重叠的 PR 进行合并；在作者确认后，将较旧的 PR 关闭并标记为 superseded。有关归属规则，请参阅 Superseding PRs。
  3. 在过期关闭窗口开始之前，将休眠的 PR 标记为 stale-candidate。
  4. 在重新打开任何已因长时间未活动而关闭的问题之前，需要重新变基并提供新的验证证据。

自动化覆盖

当自动化输出产生审查副作用时，请使用此方法：

  1. 不正确的风险标签 — 添加 risk: manual，然后设置预期的 risk:* 标签。
  2. 问题分类时自动关闭错误 — 重新打开，移除路由标签，并留下一条澄清评论。
  3. 标记垃圾或噪声 — 保留一个规范性的维护者注释，移除冗余的路由标签。
  4. PR 范围不明确 — 在深入审查之前请求拆分；不要试图同时审查两个不同的关注点。

交接

在将审查权移交其他维护者或代理时，请包含：

  1. 范围摘要。
  2. 当前风险类别及理由。
  3. 你已验证的内容。
  4. 未解决的阻塞项。
  5. 建议的下一步操作。

这有助于降低上下文丢失的风险，并避免下一位审阅者重复执行你已经完成的拉取操作。

每周队列清理

  • 遍历陈旧队列。仅对已接受但被阻塞的工作应用 no-stale。
  • 优先处理 size: XS/S 的 bug 和安全相关 PR。
  • 将重复出现的支持问题转化为文档改进和自动回复指导。

目标是建立一个队列，其中每个待处理的 PR（Pull Request）都处于以下状态之一：正在积极审查中、等待作者处理，或等待外部因素解决——绝不会因为无人处理而闲置。
