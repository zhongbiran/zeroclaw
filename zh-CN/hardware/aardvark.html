逐层

第 1 层 — aardvark-sys（USB 发送方）

文件： crates/aardvark-sys/src/lib.rs

这是唯一一层直接与原始 C 库交互的层。可以把它看作是一个轻量级的翻译器：它将 C 函数调用转换为安全的 Rust 代码。

算法：

find_devices()
  → 调用 aa_find_devices(16, buf)       // 询问 C 库有多少个适配器
  → 返回端口号列表                      // [0, 1, ...] 每个适配器对应一个

open_port(port)
  → 调用 aa_open(port)                  // 打开该特定适配器
  → 如果句柄 ≤ 0，返回 OpenFailed
  → 否则返回 AardvarkHandle{ _port: handle }

i2c_scan(handle)
  → 遍历 addr in 0x08..=0x77            // 每个有效的 7 位地址
      尝试 aa_i2c_read(addr, 1 byte)    // 敲门
      如果收到 ACK → 加入列表            // 设备已响应
  → 返回活跃地址列表

i2c_read(handle, addr, len)
  → aa_i2c_read(addr, len bytes)
  → 返回 Vec<u8> 类型的字节

i2c_write(handle, addr, data)
  → aa_i2c_write(addr, data)

spi_transfer(handle, bytes_to_send)
  → aa_spi_write(bytes)                // 全双工：发送 + 接收
  → 返回接收到的字节

gpio_set(handle, direction, value)
  → aa_gpio_direction(direction)       // 哪些引脚是输出
  → aa_gpio_put(value)                 // 设置输出电平

gpio_get(handle)
  → aa_gpio_get()                      // 读取所有引脚电平作为位掩码

Drop(handle)
  → aa_close(handle._port)            // 析构时始终关闭

在存根模式（无 SDK）下：每个方法都会立即返回 Err(NotFound)。find_devices() 返回 []。不会发生崩溃。

第 2 层 — AardvarkTransport（桥接层）

文件： crates/zeroclaw-hardware/src/aardvark.rs

ZeroClaw 的其余部分使用单一语言：ZcCommand → ZcResponse。AardvarkTransport 负责在该协议与上述 aardvark-sys 调用之间进行转换。

算法：

send(ZcCommand) → ZcResponse

  从 cmd.name 提取命令名称
  从 cmd.params（serde_json 值）提取参数

  匹配 cmd.name：

    "i2c_scan"   → 打开句柄 → 调用 i2c_scan()
                   → 将找到的地址格式化为十六进制列表
                   → 返回 ZcResponse{ output: "0x48, 0x68" }

    "i2c_read"   → 解析 addr（十六进制字符串）+ len（数字）
                   → 打开句柄 → i2c_enable(bitrate)
                   → 调用 i2c_read(addr, len)
                   → 将字节格式化为十六进制
                   → 返回 ZcResponse{ output: "0xAB 0xCD" }

    "i2c_write"  → 解析 addr + 数据字节
                   → 打开句柄 → i2c_write(addr, data)
                   → 返回 ZcResponse{ output: "ok" }

    "spi_transfer" → 解析 bytes_hex 字符串 → 解码为 Vec<u8>
                     → 打开句柄 → spi_enable(bitrate)
                     → spi_transfer(bytes)
                     → 将接收到的字节格式化为十六进制

    "gpio_set"   → 解析 direction + value 位掩码
                   → 打开句柄 → gpio_set(dir, val)
                   → 返回 ZcResponse{ output: "ok" }

    "gpio_get"   → 打开句柄 → gpio_get()
                   → 将位掩码值格式化为字符串

  遇到任何 AardvarkError → 返回 ZcResponse{ error: "..." }

关键设计选择——延迟打开： 句柄为每个命令重新打开，并在命令结束时释放。这意味着没有保持的连接、无需清理状态，也无需任何“是否仍打开”的逻辑。

第 3 层 — 工具（智能体调用的内容）

文件： crates/zeroclaw-hardware/src/aardvark_tools.rs

每个工具都是一个轻量级封装。它：

  1. 验证代理的 JSON 输入
  2. 确定要使用的物理设备
  3. 构建一个 ZcCommand
  4. 调用 AardvarkTransport.send()
  5. 以文本形式返回结果I2cScanTool.call(args)

→ 在 args 中查找 “device”（默认值：“aardvark0”） → 在注册表中查找该设备 → 构建 ZcCommand{ name: "i2c_scan", params: {} } → 发送至 AardvarkTransport → 返回 “Found: 0x48, 0x68”（或 “No devices found”）

I2cReadTool.call(args) → 要求 args["addr"] 和 args["len"] → 构建 ZcCommand{ name: "i2c_read", params: {addr, len} } → 发送 → 返回十六进制字节

I2cWriteTool.call(args) → 要求 args["addr"] 和 args["data"]（十六进制字符串或数组） → 构建 ZcCommand{ name: "i2c_write", params: {addr, data} } → 发送 → 返回 “ok” 或错误信息

SpiTransferTool.call(args) → 要求 args["bytes"]（十六进制字符串） → 构建 ZcCommand{ name: "spi_transfer", params: {bytes} } → 发送 → 返回接收到的字节

GpioAardvarkTool.call(args) → 要求 args["direction"] 和 args["value"]（设置模式） 或者无额外参数（获取模式） → 构建相应的 ZcCommand → 发送 → 返回结果

DatasheetTool.call(args) → action = args["action"]: “search” | “download” | “list” | “read” → “search”: 返回用于查找该设备的 Google/厂商搜索 URL → “download”: 从 args["url"] 获取 PDF → 保存至 ~/.zeroclaw/hardware/datasheets/ → “list”: 扫描数据手册目录 → 返回文件名 → “read”: 打开已保存的 PDF 并返回其文本内容—

第 4 层 — 设备注册表（地址簿）

文件： crates/zeroclaw-hardware/src/device.rs

注册表是连接设备的运行时映射。每个条目存储：别名、类型、功能、传输句柄。

register("aardvark", vid=0x2b76, ...)
  → DeviceKind::from_vid(0x2b76)  → DeviceKind::Aardvark
  → DeviceRuntime::from_kind()    → DeviceRuntime::Aardvark
  → 分配别名 "aardvark0"（第二个设备则为 "aardvark1"，依此类推）
  → 在 HashMap 中存储条目

attach_transport("aardvark0", AardvarkTransport, capabilities{i2c,spi,gpio})
  → 在条目中存储 Arc<dyn Transport>

has_aardvark()
  → 检查是否存在 kind == Aardvark 的条目 → true / false

resolve_aardvark_device(args)
  → 读取 "device" 参数（默认值："aardvark0"）
  → 在 HashMap 中查找别名
  → 返回 (alias, DeviceContext{ transport, capabilities })

第 5 层 — boot()（启动接线）

文件： crates/zeroclaw-hardware/src/lib.rs

boot() 在启动时运行一次。对于 Aardvark：

boot()
  ...
  aardvark_ports = aardvark_sys::AardvarkHandle::find_devices()
  // → 在存根模式下为 []，若连接了一个适配器则为 [0]

  for (i, port) in aardvark_ports:
    alias = registry.register("aardvark", vid=0x2b76, ...)
    // → "aardvark0", "aardvark1", ...

    transport = AardvarkTransport::new(port, bitrate=100kHz)
    registry.attach_transport(alias, transport, {i2c:true, spi:true, gpio:true})

    log "[registry] aardvark0 就绪 → Total Phase 端口 0"
  ...

第 6 层 — 工具注册表（加载器）

文件： crates/zeroclaw-hardware/src/tool_registry.rs

在调用 boot() 之后，工具注册表会检查当前存在的硬件，并仅加载相关的工具：

ToolRegistry::load(devices)

  # always loaded (Pico / GPIO)
  register: gpio_write, gpio_read, gpio_toggle, pico_flash, device_list, device_status

  # only loaded if an Aardvark was found at boot
  if devices.has_aardvark():
    register: i2c_scan, i2c_read, i2c_write, spi_transfer, gpio_aardvark, datasheet

这就是为什么 hardware_feature_registers_all_six_tools 测试在存根模式下仍然通过的原因——has_aardvark() 返回 false，不会加载额外的工具，计数保持为 6。
