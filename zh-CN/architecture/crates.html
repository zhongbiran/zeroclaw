ZeroClaw Docs

Crates

工作区按层级划分。Edge crates 负责与外部世界通信；core crates 负责协调；support crates 提供工具函数。每个 crate 都有独立的 rustdoc 文档——详见 API (rustdoc)。

层：核心

zeroclaw-runtime

代理循环、安全策略执行、SOP 引擎、定时任务调度器、入职向导和 TUI。依赖于所有其他核心和边缘 crate。

值得注意的子模块：

  • agent/ — 主要的请求/响应循环、流式处理、工具调用编排
  • security/ — 策略类型、沙箱检测、一次性密码（OTP）、紧急停止
  • sop/ — 标准操作程序引擎（参见 SOP → 概述）
  • onboard/ — 首次运行向导（wizard.rs）
  • memory/ — 使用运行时级缓存和合并计划包装 zeroclaw-memory
  • service/ — systemd / launchctl / Windows 服务集成

zeroclaw-config

TOML 模式及其验证。处理：

  • 自主性级别枚举（ReadOnly / Supervised / Full）
  • 加密密钥存储（本地密钥文件）
  • 工作区解析（环境变量、Homebrew 路径、XDG、容器检测）
  • 模式版本控制与迁移

所有面向用户的配置键均在 参考 → 配置 中记录，该文档由本 crate 生成。

zeroclaw-api

内核 ABI。定义了三个公共特性：

  • Provider — 具有流式传输能力标志的 LLM 客户端接口
  • Channel — 用于消息收发的通道
  • Tool — 代理可调用的功能

运行时仅依赖于这些特性，而不依赖于具体的实现。这就是为什么添加 provider/channel/tool 只需实现一个特性，而无需修补核心。

层：边缘

zeroclaw-providers

所有 LLM 客户端实现以及路由和回退包装器。请参阅 模型提供商 → 概述 获取列表。

结构：

  • traits.rs — 来自 zeroclaw-api 的重新导出以及 provider 内部辅助函数
  • anthropic.rs、openai.rs、ollama.rs……——每个原生提供程序对应一个文件
  • compatible.rs — 一个被 20 多个提供商（如 Groq、Mistral、xAI、Venice 等）复用的单一 OpenAI 兼容实现
  • router.rs — 多提供者路由器，根据任务提示进行路由
  • reliable.rs — 回退链包装器
  • streaming.rs — SSE 解析、令牌估算、工具调用增量

zeroclaw-channels

30+ 消息集成。请参阅 Channels → 概览 获取目录。

所有通道均实现了来自 zeroclaw-api 的 Channel 特性。每个通道都通过特性门控（feature-gated）进行控制——最小化构建仅包含你编译的通道。

orchestrator/ 子模块负责处理消息流式传输、草稿更新、多消息拆分以及 ACP 服务器。

zeroclaw-gateway

HTTP/WebSocket 网关。通过以下方式暴露运行时：

  • REST API（会话、内存、状态、定时任务管理）
  • 用于流式响应的 WebSocket
  • Web 仪表板（静态资源 + 身份验证）
  • Webhook 端点（来自推送的通道）

默认情况下需要配对；[gateway.allow_public_bind = true] 允许绑定到 0.0.0.0。

zeroclaw-tools

代理调用的可调用工具。不要与 CLI zeroclaw 子命令混淆。

包含：browser、http、pdf_extract、web_search、shell、file_read、file_write、hardware_probe 等。请参阅 工具 → 概述。

每个工具都通过工厂注册，并通过本地化的 Fluent 字符串向模型描述。

图层：支持

zeroclaw-memory

对话记忆与检索。SQLite 是默认后端；PostgreSQL 可通过 --features memory-postgres 启用，适用于需要共享并发写入存储的多实例部署。可选：

  • 嵌入后端（OpenAI、Ollama、本地）
  • 在存储的对话上进行向量检索（在 PostgreSQL 上使用 pgvector）
  • 记忆巩固（摘要、事实提取）

zeroclaw-tool-call-parser

模型侧的工具调用语法解析。处理不同提供商之间的差异：

  • OpenAI 风格的 tool_calls JSON
  • Anthropic 风格的 <tool_use> 块
  • Qwen/Ollama 的函数调用格式
  • 原生工具调用流式增量

zeroclaw-plugins

用于进程外工具实现的动态插件加载器。请参阅 开发 → 插件协议。

zeroclaw-hardware

硬件抽象层——GPIO、I2C、SPI、USB。平台相关。请参阅 硬件 → 概述。

zeroclaw-infra

追踪、指标、结构化日志。所有 crate 都通过此层发出事件。

zeroclaw-macros

为配置模式、工具注册和通道注册派生宏。减少工作区中的样板代码。

zeroclaw-tui

终端界面。可选 — 使用 --features tui 进行编译。

aardvark-sys, robot-kit

hardware 子模块使用的专用硬件支持。除非你正在启动特定的外设，否则不在本手册范围内。

功能标志

微内核路线图（RFC #5574）定义了一个功能标志分类体系。对用户而言，其实际影响是：

  • default — 一个合理的核心构建
  • ci-all — 全部启用，用于 CI
  • channel-<name> — 按频道选择加入（例如 channel-matrix、channel-discord）
  • provider-<name> — 按提供商选择加入
  • hardware — 启用硬件子系统
  • tui — 终端用户界面

运行 cargo metadata --format-version 1 | jq '.workspace_members' 或查看顶层 Cargo.toml 以获取完整列表。
