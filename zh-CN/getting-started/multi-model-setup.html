多模型设置与回退链

多模型提供商使用的常见模式指南：成本优化、质量分层、本地优先并附带托管回退、API 密钥轮换以及速率限制弹性。

参考材料位于提供程序系统的以下位置：

  • 模型提供商 → 概述 — 提供商的定义、配置结构
  • 模型提供商 → 回退与路由 — reliable 和 router 元提供商
  • 模型提供商 → 目录 — 每个提供商的配置结构

何时使用多模型设置

多模型配置适用于：

  • 高可靠性：当主用提供商发生故障时，自动回退到备用提供商
  • 成本优化：在速率限制场景下，通过回退链路由昂贵的模型
  • 区域弹性：使用地理分布式的提供商来处理特定区域的故障
  • 能力灵活性：当某个模型缺少所需功能（例如工具调用、视觉能力）时，尝试使用其他模型。
  • 速率限制处理：在收到 429（速率限制）响应时轮换 API 密钥
  • 开发与测试：无需更改代码即可在云端和本地模型之间切换

核心概念

回退提供程序链

当某个提供商遇到临时错误（如超时、连接失败、身份验证问题）时，ZeroClaw 会按照指定的顺序自动尝试回退到其他提供商。

示例：如果您的主要提供商是 openai，但它暂时不可用，ZeroClaw 可以自动回退到 anthropic，然后是 groq。

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]

当主提供者恢复时，ZeroClaw 会恢复使用它（无粘性故障转移）。

模型级回退

某些模型可能并非在所有地区都可用，或者当重型模型受到速率限制时，您可能希望使用更快的模型。

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6", "gpt-4o"] }

如果 claude-opus-4-7 失败或不可用，ZeroClaw 会按顺序尝试回退模型，同时保持在同一提供商内（除非也配置了提供商级别的回退）。

API 密钥轮换

对于经常遇到速率限制的服务提供商，您可以提供额外的 API 密钥，ZeroClaw 将在收到 429 响应时轮换使用这些密钥。

[reliability]
api_keys = ["sk-key-2", "sk-key-3", "sk-key-4"]

始终优先尝试主要的 api_key（全局或按通道配置）；这些额外的密钥会在遇到速率限制错误时轮换使用。

提供者重试

每个提供者的尝试都包括可配置的指数退避重试，然后再移动到下一个备用提供者。

[reliability]
provider_retries = 2          # Retry count per provider
provider_backoff_ms = 500     # Initial backoff in milliseconds

配置结构

所有多模型行为均位于 config.toml 的 [reliability] 部分。有关完整的字段索引和默认值，请参阅 配置参考。

示例配置

基本回退链

设置一个简单的回退机制，从你的主提供商到备用提供商：

default_provider = "openai"
default_model = "gpt-4o"

[reliability]
fallback_providers = ["anthropic"]

行为：如果 OpenAI 超时或返回错误，ZeroClaw 将使用指数退避策略重试两次，然后尝试使用 Anthropic 执行相同的请求。

高可靠性多提供商设置

将提供商回退与模型回退和 API 密钥轮换结合使用：

default_provider = "openai"
default_model = "gpt-4o"
api_key = "sk-openai-primary"

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]
api_keys = ["sk-openai-backup-1", "sk-openai-backup-2"]

[reliability.model_fallbacks]
"gpt-4o" = ["gpt-4-turbo", "gpt-3.5-turbo"]
"gpt-4-turbo" = ["gpt-3.5-turbo"]

行为：

  1. 尝试使用主密钥（2 次重试）调用 OpenAI gpt-4o
  2. 在速率限制时，轮换到备用 API 密钥
  3. 如果 OpenAI 仍然失败，则回退到 Anthropic，并使用相同的模型请求（Anthropic 将选择可用的等效模型）。
  4. 如果 Anthropic 不可用，请尝试 Groq，然后是 OpenRouter
  5. 如果模型不可用，请尝试按顺序使用备用模型

本地开发，云端回退

使用本地 Ollama 实例作为主服务，如果不可用则回退到云服务提供商：

default_provider = "ollama"
default_model = "llama2:70b"
api_url = "http://localhost:11434"

[reliability]
fallback_providers = ["openrouter", "groq"]

行为：如果 Ollama 宕机或超时，则自动切换至 OpenRouter 或 Groq，而无需进行任何配置更改。

成本优化：重型模型与快速回退

对于复杂任务使用昂贵的推理模型，但回退到更快的模型：

default_provider = "anthropic"
default_model = "claude-opus-4-7"

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6"] }

行为：当 Opus 受到速率限制或响应缓慢时，自动切换至 Sonnet（通常速度快 2–3 倍且成本更低）。

多区域设置

对于具有多区域部署的组织：

# 主要美国区域
default_provider = "anthropic"
default_model = "claude-sonnet-4-6"

[reliability]
# 如果美国 Anthropic 不可用，则回退到欧盟区域提供商
fallback_providers = ["bedrock"]  # 多区域的 AWS Bedrock
provider_retries = 3
provider_backoff_ms = 1000

确保每个备用提供商在您的环境中都有凭据：

export ANTHROPIC_API_KEY="..."
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."

热重载行为

[reliability] 部分支持热重载。在通道或网关运行时，对 config.toml 的更新将在下一条入站消息时生效，无需重启。

错误处理与回退触发

回退由以下原因触发：

  • 超时：提供程序未在配置的超时时间内响应
  • 连接错误：网络/DNS 故障
  • 身份验证错误：凭据无效（仅在检测到临时身份验证服务问题时重试）
  • 速率限制 (429)：HTTP 429；首先触发 API 密钥轮换，然后进行提供商回退
  • 服务不可用 (503)：临时服务问题
  • 未找到模型：如果已配置，则触发模型回退链

回退不会由以下情况触发：

  • 无效请求 (400)：输入格式错误；重试无济于事
  • 永久认证失败：无效的 API 密钥格式
  • 模型输出错误：模型已响应，但返回了错误

调试回退活动

启用运行时跟踪以调试回退行为：

[可观测性]
运行时跟踪模式 = "滚动"
运行时跟踪路径 = "state/runtime-trace.jsonl"

然后查询跟踪信息：

# 显示所有回退事件
zeroclaw doctor traces --contains 回退

# 显示提供程序重试详情
zeroclaw doctor traces --contains 提供者

# 显示速率限制轮换
zeroclaw doctor traces --contains "429"

最佳实践

  1. 按可靠性排序：在 fallback_providers 中将最可靠的提供商排在前面
  2. 测试回退链：在生产环境使用前，验证回退行为
  3. 监控 API 密钥轮换：跟踪速率限制事件，以了解轮换何时处于活动状态
  4. 保持模型回退的语义一致性：不要有意地从推理模型回退到聊天模型
  5. 使用环境变量：将敏感的 API 密钥存储在环境变量中，而不是配置文件中。
  6. 文档回退意图：在配置文件中添加注释，解释每个回退机制存在的原因
  7. 验证多模型凭据：确保所有备用提供商都设置了有效的凭据

凭证解析

每个回退提供程序使用标准解析顺序独立解析凭据：

  1. 来自配置/CLI的显式凭据
  2. 特定提供者的环境变量
  3. 通用回退：ZEROCLAW_API_KEY，然后是 API_KEY

重要：主提供者的 API 密钥不会被备用提供者自动复用。请为每个提供者单独设置凭据。

示例：

export OPENAI_API_KEY="sk-..."
export ANTHROPIC_API_KEY="claude-..."
export GROQ_API_KEY="gsk-..."

限制与约束

  • 最大回退提供商数量：受配置文件大小限制（通常支持 100 多个链）
  • 每个模型的最大回退次数：无硬性限制
  • API 密钥轮换：在超时之前会尝试所有密钥
  • 重试次数：可针对每个提供商进行配置，并采用指数退避策略
  • 总超时预算：在重试和回退中累计；通道级超时仍然适用
