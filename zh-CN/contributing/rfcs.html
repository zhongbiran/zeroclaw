RFC 流程

对 ZeroClaw 的架构、用户界面或核心策略进行重大变更前，需通过 RFC（请求评论）流程，然后再实施。该流程旨在揭示设计上的权衡，让维护者和贡献者有机会尽早提出反对意见，并留下可搜索的记录，说明做出某项决策的 原因。

治理、RFC 批准规则和投票阈值在 RFC #5577 中定义。

何时提交 RFC 与仅提交 PR

更改先提交 RFC（请求评论）吗？
新的通道实现不——请提交一个 PR
新的提供程序实现不——请提交一个 PR
新工具不——请提交一个 PR
错误修复不——请提交一个 PR
新的配置键取决于——如果它符合现有 schema 的形状，则提交 PR。如果它引入了新的子系统或范式，则提交 RFC。
更改已建立的默认值是的——RFC
破坏现有配置的架构迁移是的——RFC
影响多个 crate 的跨模块重构是的——RFC
新的子系统（例如，一个新的安全层，一个新的协议）是的——RFC
治理、发布流程或贡献模型的变更是的——RFC

经验法则：如果你在写代码前想要征求第二意见，那就是 RFC。如果很明显该构建什么，那就是 PR。

提交 RFC

RFC 是标记为 type:rfc 的 GitHub Issue。标题格式：

RFC: <short description of the proposal>

主体内容 — 根据提案的大小进行调整：

  1. 问题 — 用户痛点或系统缺陷是什么，促使了这一需求？
  2. 提案 — 你打算做什么？
  3. 设计 — 细节；代码草图、模式结构、迁移计划
  4. 考虑的替代方案 — 你还评估了哪些其他方案，以及为什么没有选择它们？
  5. 非目标 — 本提案明确不试图解决的问题
  6. 风险与缓解措施 — 可能出现的问题，以及回滚方案
  7. 发布 — 是否通过功能标志控制？是否支持模式版本控制？是否存在破坏性变更窗口？

已提交的 RFC 会进入讨论阶段（默认 7 天，较大的提案可延长）。任何人都可以发表评论。维护者会参与讨论。RFC 作者会根据反馈对文档内容进行迭代修改。

批准

根据 RFC #5577，RFC 需由三分之二以上的维护者多数通过方可生效。结果如下：

  • 已接受 — 问题已关闭，并添加了 status:accepted 标签，同时由维护者评论总结了最终方案。随后可以继续进行实现相关的 PR。
  • 已拒绝 — 问题已关闭，并添加了 status:rejected 标签及理由。该记录仍然存在；重新提议需要提出实质性的不同观点。
  • Deferred — 问题保持打开状态，状态为 status:deferred；稍后重新审视。
  • 已撤回 — 作者将其撤回。在不损害任何权利的情况下关闭。

实现已接受的 RFC

实现相关的 PR 应：

  • 参考 RFC 问题编号（Implements #5574 phase 1）
  • 符合既定的设计规范——如果在实现过程中细节发生变化，请更新 RFC 正文或提交后续澄清问题
  • 如果 RFC 要求逐步推出，则通过功能标志来发布。
  • 为受破坏性变更影响的用户提供迁移路径

大型 RFC 通常会跨越多个 PR 并在多个版本中发布。随着各个阶段的落地，RFC 的跟踪注释会进行更新。

当前开放的 RFC

开放 RFC 是了解 ZeroClaw 未来发展的最佳主要来源。浏览：

gh issue list --repo zeroclaw-labs/zeroclaw --label type:rfc --state open

截至撰写时，值得关注的开放 RFC 包括：

  • #5787 — 用 Mozilla Fluent 替换 TOML i18n（此分支为具体实现）
  • #5890 — 多代理 UX 流程设计
  • #5626 — 可观测性默认设置（策略问题：v0.8 默认是否启用 Prometheus）
  • #5934 — 文档实现跟踪（RFC #5576 的多阶段发布）

已批准的 foundational RFCs

这些内容会影响其他所有方面。在提出跨领域更改之前，请先阅读它们：

  • #5574 — 微内核过渡：crate 拆分、特性标志分类、v1.0 路径
  • #5576 — 文档标准和知识架构
  • #5577 — 项目治理：核心团队、投票阈值、本文档的权威性
  • #5579 — 工程基础设施：CI 流水线、发布自动化
  • #5615 — 贡献文化：人类/AI 共同作者规范
  • #5653 — 零妥协：错误处理、死代码策略、发布就绪标准

AI 生成的 RFC

根据 RFC #5615 的规定，明确允许由 AI 助手（在人类赞助人的支持下）撰写 RFC。如果某个 RFC 是在 AI 的帮助下起草的：

  • 在正文中明确标注（“由 Claude 起草，@singlerider 审核”）
  • 赞助人负责准确性并回应审查
  • 人类进行批准投票，而不是 AI

到目前为止，这一做法效果良好——将 AI 生成的草稿视为一等公民，但请记住，赞助者需对此负责。
