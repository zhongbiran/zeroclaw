ZeroClaw Docs

流式传输

ZeroClaw 中所有支持流式 API 的提供商都会以 token 为单位进行流式传输。运行时会将这些流转发给支持部分更新的通道适配器（如 Discord、Slack、Telegram 以及网关的 WebSocket），从而让用户在模型生成文本时实时看到内容逐步呈现。

流式传输的内容

该 provider 特性会发出 StreamEvent 值：

事件当
TextDelta(String)助手文本的新令牌
ReasoningDelta(String)推理/思维链令牌（o系列、DeepSeek-R1、Qwen-thinking）
ToolCall { name, args }模型决定调用一个工具
PreExecutedToolCall一个预执行的提供商端工具调用（例如 Gemini 地面搜索）
PreExecutedToolResult其结果
最终 { usage }流完成；令牌使用总量

通道通过 Channel 特性的出站流钩子消费这些事件。

功能标志

一个提供程序暴露了两个标志，以便运行时知道它可以期望什么：

#![allow(unused)]
fn main() {
fn supports_streaming(&self) -> bool { true }
fn supports_streaming_tool_events(&self) -> bool { true }
}
  • supports_streaming — 对于所有正在积极维护的提供商，该值为 true
  • supports_streaming_tool_events — 当提供者在流式传输期间发出 ToolCall 事件（而不是在结束时）时，此值为 true

OpenAI 兼容提供商的行为存在差异：有些提供商会逐块流式传输工具调用参数增量，而另一些提供商仅在调用完成后才发出调用。compatible.rs 中的 SSE 解析器能够处理这两种情况。

通道侧流式传输

通道会宣传其自身的流媒体功能：

#![allow(unused)]
fn main() {
fn supports_draft_updates(&self) -> bool;           // 就地编辑消息
fn supports_multi_message_streaming(&self) -> bool; // 将一条回复拆分为多条消息
}
通道草稿更新多消息
命令行界面✓—
Discord✓✓
Slack✓✓
Telegram✓部分
矩阵✓—
Mattermost✓—
电子邮件——
短信 / 语音——
网关（WebSocket）✓✓

当提供者和通道均支持流式传输时，流程如下：提供者发出 TextDelta → 运行时将其传递给通道 → 通道编辑已发送的消息。编辑频率受通道配置中的 draft_update_interval_ms 限制（默认值：500 毫秒），以避免触发速率限制。

推理块

推理模型（OpenAI o 系列、DeepSeek-R1、Qwen-thinking 变体）会发出与常规文本分离的 ReasoningDelta 事件。默认情况下，运行时会将这些事件从出站流中剥离——参见 crates/zeroclaw-channels/src/orchestrator/mod.rs 中的 <think>…</think> 处理逻辑。用户看到的是最终答案，而非思维链。

向用户展示推理过程：

[channels.<name>]
show_reasoning = true

默认情况下，此功能处于关闭状态，因为推理内容（a）通常冗长，且（b）有时会暴露内部思考过程，这可能会让最终用户感到困惑。

在具备推理能力的模型上完全禁用推理：

[providers.models.<name>]
think = false
reasoning_effort = "none"

这两个字段都是顶层字段；右侧的名称取决于提供商/端点。同时设置这两个字段可以覆盖 Ollama 原生、Ollama OpenAI 兼容以及支持 reasoning_effort 的上游 API。

中途工具调用

当模型决定调用工具时，提供者会发出 ToolCall。运行时：

  1. 暂停从提供者的流中读取
  2. 将任何缓冲的文本刷新到通道
  3. 运行该工具（受安全验证约束 — 详见 安全 → 概述）
  4. 恢复对话，并附加工具结果
  5. 为下一个助手回合打开一个新的流式调用

从用户的角度来看：文本，然后是一个可见的指示器，表明代理通过特定于频道的提示运行了工具，然后是更多文本。对于没有输入指示器的频道，工具调用与下一个文本块之间的间隙是唯一的信号。

非流式提供商

如果某个提供商一次性返回整个响应（较旧的 OpenAI 兼容端点、旧版 Gemini），运行时将合成一个包含完整回复的 TextDelta，后跟 Final。通道适配器仍然可以正常工作——只是它们不会看到部分响应。
