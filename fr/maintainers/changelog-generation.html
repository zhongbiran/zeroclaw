Génération du journal des modifications

La procédure officielle pour assembler CHANGELOG-next.md entre les versions stables. Cette page est chargée par la compétence changelog-generation et lue par les mainteneurs qui exécutent une version manuellement — les deux consomment le même protocole.

The release workflows (release-stable-manual.yml) automatically use CHANGELOG-next.md as the GitHub Release body if it’s at the repo root when a release fires. After the stable release ships, CHANGELOG-next.md is intentionally left on master; the next release cycle overwrites it with a fresh file. No manual cleanup is needed.

1. Définir la plage de commits

Par défaut : dernière balise stable → HEAD

PREV_TAG=$(git tag --sort=-creatordate | grep -vE '\-beta\.' | head -1)
echo "Plage : ${PREV_TAG}..HEAD"

Plage spécifiée par l’utilisateur

Accepter l’un des éléments suivants et normaliser en <from>..<to> :

EntréeInterprétation
v0.7.2v0.7.2..HEAD
v0.7.1..v0.7.2Exactement comme donné
v0.7.1 v0.7.2v0.7.1..v0.7.2

Vérifiez que les deux refs existent avant de continuer :

git rev-parse --verify <ref> 2>/dev/null || echo « ERREUR : référence introuvable »

2. Collecter et catégoriser les commits

Collecter

git log <from>..<to> --pretty=format:"%H %h %s" --no-merges

Enregistrer les SHAs complets pour l’étape de résolution des contributeurs :

git log <from>..<to> --pretty=format:"%H" --no-merges > /tmp/zc-commits.txt

Catégoriser

Associez chaque commit à une section en fonction de son préfixe de commit conventionnel. Les commits sans préfixe reconnu doivent toujours être lus et catégorisés par leur contenu — ne les ignorez jamais silencieusement.

PréfixeSection
feat:, feat(*)Quoi de neuf
fix:, fix(*)Corrections de bugs
refactor:, perf:Quoi de neuf (groupe comme « Améliorations »)
security:, fix(*security*)Quoi de neuf → Sécurité
docs:, docs(*)Quoi de neuf → Documentation (omettez les corrections de fautes de frappe triviales)
chore:, ci:, build:Omettre sauf si visible par l’utilisateur (nouveau chemin d’installation, plateforme supprimée, etc.)
breaking: ou suffixe !Modifications incompatibles — toujours afficher
Aucun préfixeLire le corps ; catégoriser par contenu ; noter dans la revue

Ordre des sections dans le fichier de sortie

  1. Préambule
  2. Points forts
  3. Quoi de neuf
  4. Corrections de bugs
  5. Modifications incompatibles (à omettre si vide)
  6. Contributeurs

3. Résolution des contributeurs

N’utilisez pas uniquement git log --pretty=format:"%an" — il ne prend pas en compte les personnes listées dans les trailers Co-Authored-By. Utilisez le champ GraphQL authors de GitHub, qui résout les auteurs directs et les co-auteurs.

Requête

Paginez par lots de 100 commits. Utilisez pageInfo.endCursor tant que hasNextPage est true.

{
  repository(owner: "zeroclaw-labs", name: "zeroclaw") {
    ref(qualifiedName: "refs/heads/master") {
      target {
        ... on Commit {
          history(first: 100) {
            pageInfo { hasNextPage endCursor }
            nodes {
              oid
              authors(first: 10) {
                nodes {
                  user { login }
                  email
                }
              }
            }
          }
        }
      }
    }
  }
}

Exécuter via gh :

gh api graphql -f query='<query>'

Filtrer la liste — exclure tout ce qui suit

Par motif de connexion :

  • Toute connexion se terminant par [bot]
  • web-flow
  • dependabot
  • github-actions
  • blacksmith

Par motif d’e-mail :

  • *@noreply.github.com
  • *@noreply.anthropic.com
  • *noreply*

Noms de modèles d’IA apparaissant comme noms d’auteurs (pas des identifiants) :

  • Claude, Copilot, ChatGPT, Codex, Gemini, GitHub Copilot
  • Tout nom correspondant à ^(gpt|claude|gemini|copilot)-

Sortie

Croisez chaque oid de la réponse GraphQL avec /tmp/zc-commits.txt pour ne conserver que les commits dans la plage de la version. Collectez les identifiants uniques, triez-les de manière insensible à la casse, et préfixez chacun par @.

4. Format du fichier CHANGELOG-next.md

Préambule

Cette version majeure introduit des améliorations significatives qui simplifient l’utilisation du produit pour tous les utilisateurs. Elle inclut également de nouvelles fonctionnalités conçues pour répondre aux besoins des professionnels et des particuliers.

Points forts

Quatre à six puces. Commencez par l’impact visible par l’utilisateur, et non par les détails d’implémentation. Chaque puce doit répondre à : « Que puis-je faire maintenant que je ne pouvais pas faire avant ? » ou « Qu’est-ce qui vient de s’améliorer ? »

Quoi de neuf

Regrouper les entrées par zone. Utiliser uniquement les groupes qui contiennent du contenu.

Groupes suggérés (ajoutez ou supprimez librement) :

  • Architecture & Espace de travail
  • Agent et Runtime
  • Fournisseurs
  • Chaînes
  • Outils
  • Configuration
  • Tableau de bord Web
  • Compétences
  • Sécurité
  • Matériel
  • Installation et distribution
  • Dépendances et avis de sécurité

Write each entry as a sentence for a human reader — not a raw commit message. Reference PR numbers with (#NNNN) where available.

Corrections de bugs

Un tableau récapitulatif. Colonnes : Zone | Correction. Regroupez plusieurs corrections pour une même fonctionnalité dans une seule ligne lorsque cela est plus clair que des lignes séparées.

Modifications incompatibles

Signalez chaque changement majeur avec un chemin de migration. Recherchez :

  • Modifications du schéma de configuration (champs renommés ou supprimés)
  • Sous-commandes ou indicateurs CLI obsolètes ou renommés
  • Modifications de la limite du crate ou de la surface de l’API publique
  • Modifications de comportement derrière des clés de configuration existantes

S’il n’y a pas de modifications incompatibles, omettez entièrement cette section.

Contributeurs

@login gère les noms d’utilisateur de l’étape 3, triés de manière insensible à la casse, un par ligne.

Pied de page

*Full diff: `git log <prev-tag>..<next-tag> --oneline`*

5. Intégration du flux de travail de sortie et de libération

Écrire l’emplacement

Écrire dans CHANGELOG-next.md à la racine du dépôt — c’est le chemin que les workflows de publication recherchent. Une copie se trouve également dans tmp/CHANGELOG-next.md pour une révision en session avant la validation.

Valider

git add CHANGELOG-next.md
git commit -m chore(release): ajouter CHANGELOG-next.md pour vX.Y.Z

Remplacez vX.Y.Z par la version de la prochaine publication. Demandez la confirmation à l’utilisateur avant de valider.

Pousser

Pousser vers la branche PR de la version ouverte sur zeroclaw-labs/zeroclaw :

git push upstream <branch>

Ne poussez pas directement sur master.

Consommation de workflow

release-stable-manual.yml vérifie la présence de CHANGELOG-next.md au début du job de release. Si ce fichier est trouvé, son contenu devient le corps de la GitHub Release. Sinon, le workflow utilise par défaut les notes générées automatiquement, limitées aux commits de type feat:.

After a successful stable release, CHANGELOG-next.md is intentionally left on master. The next release cycle will overwrite it. No manual cleanup is required.
