Aardvark

Un guide en langage clair de chaque élément et de leur interconnexion.

La vue d’ensemble

┌──────────────────────────────────────────────────────────────┐
│                        DÉMARRAGE (boot)                        │
│                                                              │
│  1. Demander à aardvark-sys : « des adaptateurs sont-ils branchés ? » │
│  2. Pour chacun trouvé → enregistrer un périphérique + transport │

└──────────────────────────────────────────┬───────────────────┘
                                           │
                    ┌──────────────────────▼──────────────────────┐
                    │              EXÉCUTION (boucle agent)            │
                    │                                              │
                    │  Utilisateur : « scanner le bus i2c »                        │

                    │     → la réponse revient sous forme de texte            │

Couche par couche

Couche 1 — aardvark-sys (l’émetteur USB)

Fichier : crates/aardvark-sys/src/lib.rs

C’est la seule couche qui interagit directement avec la bibliothèque C brute. Considérez-la comme un traducteur léger : elle transforme les appels de fonctions C en Rust sécurisé.

Algorithme :

find_devices()
  → appeler aa_find_devices(16, buf)       // demander à la bibliothèque C combien d'adaptateurs sont disponibles
  → retourner un Vec de numéros de port    // [0, 1, ...] un par adaptateur

open_port(port)
  → appeler aa_open(port)                  // ouvrir cet adaptateur spécifique
  → si handle ≤ 0, retourner OpenFailed
  → sinon retourner AardvarkHandle{ _port: handle }

i2c_scan(handle)
  → pour addr dans 0x08..=0x77            // chaque adresse 7 bits valide
      essayer aa_i2c_read(addr, 1 octet)   // sonner à la porte
      si ACK → ajouter à la liste          // l'appareil a répondu
  → retourner la liste des adresses actives

i2c_read(handle, addr, len)
  → aa_i2c_read(addr, len octets)
  → retourner les octets sous forme de Vec<u8>

i2c_write(handle, addr, data)
  → aa_i2c_write(addr, data)

spi_transfer(handle, bytes_to_send)
  → aa_spi_write(bytes)                    // duplex intégral : envoie et reçoit
  → retourner les octets reçus

gpio_set(handle, direction, value)
  → aa_gpio_direction(direction)           // quels pins sont des sorties
  → aa_gpio_put(value)                     // définir les niveaux de sortie

gpio_get(handle)
  → aa_gpio_get()                          // lire tous les niveaux de pins comme un masque de bits

Drop(handle)
  → aa_close(handle._port)                 // toujours fermer lors du drop

En mode stub (sans SDK) : chaque méthode renvoie immédiatement Err(NotFound). find_devices() renvoie []. Aucun plantage.

Couche 2 — AardvarkTransport (le pont)

Fichier : crates/zeroclaw-hardware/src/aardvark.rs

Le reste de ZeroClaw utilise un seul langage : ZcCommand → ZcResponse. AardvarkTransport assure la traduction entre ce protocole et les appels aardvark-sys mentionnés ci-dessus.

Algorithme :

send(ZcCommand) → ZcResponse

  extraire le nom de la commande depuis cmd.name
  extraire les paramètres depuis cmd.params (valeurs serde_json)

  correspondre à cmd.name :

    "i2c_scan"   → ouvrir le handle → appeler i2c_scan()
                   → formater les adresses trouvées sous forme de liste hexadécimale
                   → retourner ZcResponse{ output: "0x48, 0x68" }

    "i2c_read"   → analyser addr (chaîne hexadécimale) + len (nombre)
                   → ouvrir le handle → i2c_enable(bitrate)
                   → appeler i2c_read(addr, len)
                   → formater les octets en hexadécimal
                   → retourner ZcResponse{ output: "0xAB 0xCD" }

    "i2c_write"  → analyser addr + les octets de données
                   → ouvrir le handle → i2c_write(addr, data)
                   → retourner ZcResponse{ output: "ok" }

    "spi_transfer" → analyser la chaîne bytes_hex → décoder en Vec<u8>
                     → ouvrir le handle → spi_enable(bitrate)
                     → spi_transfer(bytes)
                     → retourner les octets reçus en hexadécimal

    "gpio_set"   → analyser les masques de bits direction + valeur
                   → ouvrir le handle → gpio_set(dir, val)
                   → retourner ZcResponse{ output: "ok" }

    "gpio_get"   → ouvrir le handle → gpio_get()
                   → retourner la valeur du masque de bits sous forme de chaîne

  en cas d'erreur AardvarkError → retourner ZcResponse{ error: "..." }

Choix de conception clé — ouverture paresseuse : Le descripteur est ouvert à neuf pour chaque commande et fermé à la fin. Cela signifie qu’il n’y a pas de connexion maintenue, pas d’état à nettoyer, et aucune logique du type « est-il encore ouvert ? » n’est nécessaire.

Couche 3 — Outils (ce que l’agent appelle)

Fichier : crates/zeroclaw-hardware/src/aardvark_tools.rs

Chaque outil est un fin wrapper. Il :

  1. Valide l’entrée JSON de l’agent
  2. Détermine quel appareil physique utiliser
  3. Crée un ZcCommand
  4. Appelle AardvarkTransport.send()
  5. Renvoie le résultat sous forme de texte
I2cScanTool.call(args)
  → rechercher « device » dans args (par défaut : "aardvark0")
  → trouver cet appareil dans le registre
  → construire ZcCommand{ name: "i2c_scan", params: {} }
  → envoyer à AardvarkTransport
  → retourner "Found: 0x48, 0x68" (ou "No devices found")

I2cReadTool.call(args)
  → exiger args["addr"] et args["len"]
  → construire ZcCommand{ name: "i2c_read", params: {addr, len} }
  → envoyer → retourner les octets en hexadécimal

I2cWriteTool.call(args)
  → exiger args["addr"] et args["data"] (hex ou tableau)
  → construire ZcCommand{ name: "i2c_write", params: {addr, data} }
  → envoyer → retourner "ok" ou une erreur

SpiTransferTool.call(args)
  → exiger args["bytes"] (chaîne hexadécimale)
  → construire ZcCommand{ name: "spi_transfer", params: {bytes} }
  → envoyer → retourner les octets reçus

GpioAardvarkTool.call(args)
  → exiger args["direction"] + args["value"]  (définir)
         OU aucun argument supplémentaire       (obtenir)
  → construire la commande ZcCommand appropriée
  → envoyer → retourner le résultat

DatasheetTool.call(args)
  → action = args["action"] : "search" | "download" | "list" | "read"
  → "search" :   retourner une URL de recherche Google/fournisseur pour l'appareil
  → "download" : télécharger le PDF depuis args["url"] → enregistrer dans ~/.zeroclaw/hardware/datasheets/
  → "list" :     analyser le répertoire des datasheets → retourner les noms de fichiers
  → "read" :     ouvrir un PDF enregistré et retourner son texte

Couche 4 — Registre des appareils (le carnet d’adresses)

Fichier : crates/zeroclaw-hardware/src/device.rs

Le registre est une carte d’exécution de tous les appareils connectés. Chaque entrée stocke : alias, type, capacités, gestionnaire de transport.

register("aardvark", vid=0x2b76, ...)
  → DeviceKind::from_vid(0x2b76)  → DeviceKind::Aardvark
  → DeviceRuntime::from_kind()    → DeviceRuntime::Aardvark
  → assigner l'alias "aardvark0" (puis "aardvark1" pour le second, etc.)
  → stocker l'entrée dans HashMap

attach_transport("aardvark0", AardvarkTransport, capabilities{i2c,spi,gpio})
  → stocker Arc<dyn Transport> dans l'entrée

has_aardvark()
  → vérifier si une entrée a kind == Aardvark  → true / false

resolve_aardvark_device(args)
  → lire le paramètre "device" (par défaut : "aardvark0")
  → rechercher l'alias dans HashMap
  → retourner (alias, DeviceContext{ transport, capabilities })

Couche 5 — boot() (initialisation au démarrage)

Fichier : crates/zeroclaw-hardware/src/lib.rs

boot() s’exécute une fois au démarrage. Pour Aardvark :

boot()
  ...
  aardvark_ports = aardvark_sys::AardvarkHandle::find_devices()
  // → [] en mode stub, [0] si un adaptateur est branché

  for (i, port) in aardvark_ports:
    alias = registry.register("aardvark", vid=0x2b76, ...)
    // → "aardvark0", "aardvark1", ...

    transport = AardvarkTransport::new(port, bitrate=100kHz)
    registry.attach_transport(alias, transport, {i2c:true, spi:true, gpio:true})

    log "[registry] aardvark0 prêt → Port Total Phase 0"
  ...

Couche 6 — Registre des outils (le chargeur)

Fichier : crates/zeroclaw-hardware/src/tool_registry.rs

Après boot(), le registre des outils vérifie le matériel disponible et charge uniquement les outils pertinents :

ToolRegistry::load(devices)

  # always loaded (Pico / GPIO)
  register: gpio_write, gpio_read, gpio_toggle, pico_flash, device_list, device_status

  # only loaded if an Aardvark was found at boot
  if devices.has_aardvark():
    register: i2c_scan, i2c_read, i2c_write, spi_transfer, gpio_aardvark, datasheet

C’est pourquoi le test hardware_feature_registers_all_six_tools continue de réussir en mode stub — has_aardvark() retourne false, 0 outil supplémentaire est chargé, le compteur reste à 6.

      ▼
