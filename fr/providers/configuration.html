Configuration du fournisseur

Chaque fournisseur de modèle est déclaré sous [providers.models.<nom>] dans ~/.zeroclaw/config.toml. Le <nom> est votre propre alias — c’est ainsi que vous référencez le fournisseur ailleurs dans la configuration (default_model = "claude", fallback_providers = ["claude", "local"], etc.).

Forme minimale

[providers.models.<name>]
kind = "<provider-kind>"   # requis — sélectionne l'implémentation
model = "<model-id>"       # requis — transmis au fournisseur

Presque tous les fournisseurs acceptent également :

api_key = "..."            # ou un espace réservé de variable d'environnement
base_url = "https://..."   # pour les points de terminaison compatibles OpenAI ou auto-hébergés

Identifiants

Quatre façons de fournir des identifiants, dans l’ordre de résolution :

  1. Inline api_key = "..." dans l’entrée de configuration (acceptable pour le développement, risqué pour les configurations versionnées)
  2. Stockage des secrets au niveau de la configuration — chiffré dans ~/.zeroclaw/secrets via un fichier de clé local
  3. Variable d’environnement spécifique au fournisseur — ANTHROPIC_API_KEY, ANTHROPIC_OAUTH_TOKEN, OPENAI_API_KEY, OPENROUTER_API_KEY, GROQ_API_KEY, etc.
  4. Défaut générique — ZEROCLAW_API_KEY, API_KEY

L’assistant de configuration écrit les identifiants dans le magasin de secrets par défaut. Les fichiers de configuration que vous validez ne doivent ni utiliser de clés intégrées ni d’entrées env_passthrough qui exposent les clés utilisateur.

Authentification OAuth et authentification par abonnement

Plusieurs fournisseurs prennent en charge des jetons de type OAuth / abonnement au lieu de clés API brutes :

  • Anthropic — Les jetons OAuth sk-ant-oat-* fonctionnent partout où une clé API est utilisée. Aucun coût si vous êtes sur un plan Pro/Team.
  • GitHub Copilot — authentifiez-vous via l’assistant de configuration (zeroclaw onboard) qui gère le flux OAuth. Le jeton est stocké dans le backend de secrets.
  • Gemini CLI — utilise l’authentification existante de la CLI gemini.
  • Claude Code — utilise votre connexion Claude Code.

Remplacements adaptés aux conteneurs

L’assistant de prise en main détecte Docker/Podman/Kubernetes et remplace localhost par des noms d’hôte adaptés aux conteneurs :

[providers.models.local]
kind = "ollama"
base_url = "http://host.docker.internal:11434"   # était "http://localhost:11434" sur l'hôte

Vous pouvez également forcer cela manuellement à l’exécution :

ZEROCLAW_OLLAMA_BASE_URL=http://ollama:11434 zeroclaw agent

Paramètres spécifiques à chaque fournisseur

Au-delà des champs universels, certains fournisseurs acceptent des champs supplémentaires. Points saillants :

Ollama

[providers.models.local]
kind = "ollama"
base_url = "http://localhost:11434"
model = "qwen3.6:35b-a3b"
think = false                # désactiver le mode de raisonnement pour une sortie plus rapide
reasoning_effort = "none"    # même intention, passé en tant que champ de niveau supérieur
options = { temperature = 0, num_ctx = 32768 }

Compatible avec OpenAI

[providers.models.groq]
kind = "openai-compatible"
base_url = "https://api.groq.com/openai"
model = "llama-3.3-70b-versatile"
api_key = "gsk_..."
# Facultatif — fournit des indications de streaming d'appels d'outils SSE que le point de terminaison comprend
native_tool_streaming = true

Azure OpenAI

[providers.models.azure]
kind = "azure-openai"
base_url = "https://my-resource.openai.azure.com"
deployment = "gpt-4o"
api_version = "2024-10-01-preview"
api_key = "..."

Sélection de la valeur par défaut

default_provider = "claude"
default_model    = "claude-haiku-4-5-20251001"

Les deux sont lus au démarrage de l’agent. Les canaux, les outils et les SOP peuvent être remplacés par requête.
