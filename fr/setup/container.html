ZeroClaw Docs

Docker & Conteneurs

Exécutez ZeroClaw dans Docker, Podman, Kubernetes ou n’importe quel runtime OCI.

Images officielles

Pushed to GitHub Container Registry (ghcr.io) on every stable release:

  • ghcr.io/zeroclaw-labs/zeroclaw:latest — latest stable
  • ghcr.io/zeroclaw-labs/zeroclaw:v0.7.5 — pinned
  • ghcr.io/zeroclaw-labs/zeroclaw:debian — Debian-based image (larger, broader glibc support)

Multi-arch : linux/amd64, linux/arm64.

Note on shell access: The default latest image is intentionally distroless and does not include sh, ash, or bash. Use the debian tag if you need a shell inside the container (for example, to run docker exec for debugging).

Exécution minimale

docker run -d \
  --name zeroclaw \
  -v zeroclaw-data:/zeroclaw-data \
  -p 42617:42617 \
  ghcr.io/zeroclaw-labs/zeroclaw:latest

L’image attend un état persistant à /zeroclaw-data. Lors de la première exécution, elle initialise une configuration par défaut — vous devez toujours effectuer l’onboarding avant qu’elle ne soit utile :

docker exec -it zeroclaw zeroclaw onboard

Composer

Un docker-compose.yml minimal :

services:
  zeroclaw:
    image: ghcr.io/zeroclaw-labs/zeroclaw:latest
    redémarrer: sauf-arrêt
    ports:
      - "42617:42617"      # passerelle
    volumes:
      - ./data:/zeroclaw-data
    environnement:
      ZEROCLAW_ALLOW_PUBLIC_BIND: « 1 »   # uniquement si la passerelle doit être accessible sur le LAN

After the container starts, run onboarding:

docker compose exec zeroclaw zeroclaw onboard

Supprimez ZEROCLAW_ALLOW_PUBLIC_BIND si vous avez uniquement besoin d’un accès local.

Configuration à l’intérieur des conteneurs

L’image attend la configuration à /zeroclaw-data/.zeroclaw/config.toml. Montez votre configuration locale dans :

docker run -d --name zeroclaw \
  -v $(pwd)/my-config.toml:/zeroclaw-data/.zeroclaw/config.toml:ro \
  -v zeroclaw-state:/zeroclaw-data/workspace \
  -p 42617:42617 \
  ghcr.io/zeroclaw-labs/zeroclaw:latest

Pour les charges de travail dans des conteneurs, l’assistant d’intégration détecte Docker/Podman/Kubernetes et remplace les références à localhost dans la configuration par host.docker.internal (pour Docker) ou d’autres alias adaptés aux conteneurs.

Les canaux qui interrogent (Telegram, e-mail) — fonctionnent simplement

Les canaux initiés en sortie ne nécessitent aucune configuration spéciale du conteneur. Telegram en mode sondage, IMAP, MQTT, les relais Nostr — tous utilisent le mode « pull » ; le conteneur a uniquement besoin d’une sortie (egress).

Les canaux qui reçoivent des webhooks — ont besoin d’ingress

Discord, Slack, GitHub et la plupart des canaux webhook nécessitent un accès HTTP entrant. Deux options :

  1. Exposer la passerelle — -p 42617:42617 + un proxy inverse avec TLS en amont, pointez l’URL du webhook vers l’adresse publique
  2. Utiliser un tunnel — ngrok, Cloudflare Tunnel ou Tailscale Funnel ; définissez l’URL du tunnel comme cible du webhook

L’étape de tunnel de l’assistant de prise en main gère ngrok et Cloudflare directement.

Kubernetes

Les chartes Helm et les modèles du marketplace sont publiés dans le dépôt zeroclaw-templates. Fragment de manifeste typique :

apiVersion: apps/v1
kind: Déploiement
métadonnées:
  nom: zeroclaw
spécification:
  répliques: 1
  stratégie:
    type: Recreate         # ZeroClaw est une instance unique par espace de travail
  template:
    spécification:
      conteneurs:
        - nom: zeroclaw
          image: ghcr.io/zeroclaw-labs/zeroclaw:v0.7.5
          ports:
            - `containerPort`: 42617
          volumeMounts:
            - nom: données
              chemin de montage: /zeroclaw-data
          env:
            - nom: ZEROCLAW_ALLOW_PUBLIC_BIND
              valeur: "1"
      volumes:
        - nom: données
          persistentVolumeClaim:
            nom de la revendication: zeroclaw-data

Mise à l’échelle : ZeroClaw est conçu pour un seul écriture par espace de travail. Ne mettez pas à l’échelle horizontalement — exécutez une instance par agent.

Re-authenticating after logout

If you log out of the web UI while running in a container, the existing paircode becomes invalid. Generate a new one to log back in:

docker exec -it zeroclaw zeroclaw gateway get-paircode --new

For Compose deployments, use docker compose exec instead:

docker compose exec zeroclaw zeroclaw gateway get-paircode --new

Pièges

  • Particularités du nom d’hôte sous macOS (Docker Desktop, colima, Rancher Desktop). host.docker.internal fonctionne nativement sur Docker Desktop pour macOS. Sur colima, il n’est accessible que si vous avez lancé l’installation avec colima start --network-address (sinon, le conteneur ne peut pas atteindre l’hôte — connectez-vous via l’adresse IP de la passerelle de la machine virtuelle, généralement 192.168.5.2, ou utilisez un tunnel à travers un réseau partagé). Rancher Desktop se comporte comme Docker Desktop pour les versions récentes, mais des échecs de résolution de host.docker.internal ont été signalés sur les versions plus anciennes. Si les appels au fournisseur échouent avec une erreur connection refused vers host.docker.internal, vérifiez avec docker run --rm alpine getent hosts host.docker.internal — une sortie vide signifie que le nom d’hôte n’est pas résolvable et qu’une adresse IP explicite est nécessaire.
  • Services côté hôte. Si le fournisseur est Ollama sur l’hôte, base_url = "http://host.docker.internal:11434" fonctionne sur Docker Desktop. Sous Linux Docker, vous devrez peut-être utiliser --add-host=host.docker.internal:host-gateway.
  • Persistance de la mémoire. Le fichier SQLite en mémoire se trouve dans /zeroclaw-data/workspace/. Si vous ne montez pas ce volume, chaque redémarrage entraîne la perte de l’historique des conversations.
  • Bind-mounting /zeroclaw-data. A host bind mount on /zeroclaw-data replaces the entire image directory, including the default config.toml and (previously) the dashboard bundle. The dashboard is now installed at /usr/share/zeroclawlabs/web/dist — outside the mount — so a bind mount no longer hides it. On first run, mount an empty host directory and the container bootstraps a fresh config; the gateway auto-detects the dashboard from its image path.
  • Aucun transfert matériel par défaut. GPIO / USB nécessitent des indicateurs --device explicites (--device /dev/ttyUSB0), et l’utilisateur du conteneur doit avoir le GID correspondant pour les groupes dialout/gpio.
