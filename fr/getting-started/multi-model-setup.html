Configuration multi-modèles et chaînes de secours

Un guide des modèles courants pour utiliser plusieurs fournisseurs de modèles : optimisation des coûts, niveaux de qualité, approche locale avec fallback hébergé, rotation des clés API et résilience aux limites de taux.

Les références pour le système de fournisseur se trouvent dans :

  • Fournisseurs de modèles → Aperçu — ce que sont les fournisseurs, forme de la configuration
  • Proveurs de modèles → Fallback et routage — méta-proveurs reliable et router
  • Providers de modèles → Catalogue — la forme de la configuration de chaque fournisseur

Quand utiliser une configuration multi-modèle

La configuration multi-modèles est utile pour :

  • Haute fiabilité : Bascule automatiquement vers des fournisseurs alternatifs lorsque le principal échoue
  • Optimisation des coûts : Acheminer les modèles coûteux via des chaînes de repli pour les scénarios limités en taux
  • Résilience régionale : Utilisez des fournisseur répartis géographiquement pour gérer les pannes spécifiques à une région
  • Flexibilité des capacités : Essayez différents modèles lorsque l’un manque de fonctionnalités requises (par exemple, l’appel d’outils, la vision)
  • Gestion des limites de taux : Alterner entre les clés API en cas de réponses 429 (limite de taux)
  • Développement et tests : Passez des modèles cloud aux modèles locaux sans modifier le code.

Notions fondamentales

Chaînes de fournisseurs de secours

Lorsqu’un fournisseur rencontre une erreur transitoire (délai d’attente dépassé, échec de connexion, problème d’authentification), ZeroClaw tente automatiquement les fournisseurs de repli dans l’ordre spécifié.

Exemple : Si votre principal fournisseur est openai mais qu’il est temporairement indisponible, ZeroClaw peut automatiquement basculer vers anthropic, puis groq.

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]

Lorsque le fournisseur principal se rétablit, ZeroClaw reprend son utilisation (pas de basculement persistant).

Fallbacks au niveau du modèle

Certains modèles peuvent ne pas être disponibles dans toutes les régions, ou vous pourriez souhaiter utiliser un modèle plus rapide lorsqu’un modèle lourd est soumis à des limitations de débit.

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6", "gpt-4o"] }

Si claude-opus-4-7 échoue ou est indisponible, ZeroClaw essaie les modèles de repli dans l’ordre tout en restant dans le même fournisseur (sauf si un repli au niveau du fournisseur est également configuré).

Rotation de la clé API

Pour les fournisseurs qui rencontrent fréquemment des limites de taux, vous pouvez fournir des clés API supplémentaires que ZeroClaw fera varier en cas de réponses 429.

[rélationnabilité]
api_keys = ["sk-key-2", "sk-key-3", "sk-key-4"]

La clé api_key principale (configurée globalement ou par canal) est toujours testée en premier ; ces clés supplémentaires sont rotatives en cas d’erreurs de limitation de débit.

Nombre de tentatives du fournisseur

Chaque tentative du fournisseur inclut des tentatives de nouvelle connexion configurables avec un backoff exponentiel avant de passer au prochain secours.

[reliability]
provider_retries = 2          # Retry count per provider
provider_backoff_ms = 500     # Initial backoff in milliseconds

Structure de configuration

Tout le comportement relatif aux modèles multiples se trouve dans la section [reliability] de config.toml. Consultez la documentation de référence de la configuration pour consulter la liste complète des champs et leurs valeurs par défaut.

Exemples de configurations

Chaîne de repli de base

Configurez une solution de secours simple depuis votre fournisseur principal vers une solution de secours :

default_provider = "openai"
default_model = "gpt-4o"

[reliability]
fallback_providers = ["anthropic"]

Comportement : Si OpenAI expire ou renvoie une erreur, ZeroClaw réessaiera deux fois avec une backoff exponentielle, puis tentera d’exécuter la même requête via Anthropic.

Configuration multi-fournisseuse à haute fiabilité

Combiner les sauts de secours des fournisseurs avec les sauts de secours des modèles et la rotation des clés API :

default_provider = "openai"
default_model = "gpt-4o"
api_key = "sk-openai-primary"

[reliability]
fallback_providers = ["anthropic", "groq", "openrouter"]
api_keys = ["sk-openai-backup-1", "sk-openai-backup-2"]

[reliability.model_fallbacks]
"gpt-4o" = ["gpt-4-turbo", "gpt-3.5-turbo"]
"gpt-4-turbo" = ["gpt-3.5-turbo"]

Comportement :

  1. Essayer OpenAI gpt-4o avec la clé principale (2 tentatives)
  2. En cas de limitation de taux, basculer vers les clés API de secours.
  3. Si OpenAI échoue toujours, revenez à Anthropic avec la même requête de modèle (Anthropic sélectionnera l’équivalent disponible)
  4. Si Anthropic est indisponible, essayez Groq, puis OpenRouter
  5. Si le modèle n’est pas disponible, essayez les modèles de repli dans l’ordre

Développement local avec secours via le cloud

Utiliser une instance locale d’Ollama comme principale et faire appel à un fournisseur cloud en tant que solution de secours :

default_provider = "ollama"
default_model = "llama2:70b"
api_url = "http://localhost:11434"

[reliability]
fallback_providers = ["openrouter", "groq"]

Comportement : Si Ollama est indisponible ou dépasse les délais, basculer automatiquement vers OpenRouter ou Groq sans modification de configuration.

Optimisation des coûts : modèle lourd avec dégradation rapide

Utilisez un modèle de raisonnement coûteux pour les tâches complexes, mais revenez à un modèle plus rapide :

default_provider = "anthropic"
default_model = "claude-opus-4-7"

[reliability]
model_fallbacks = { "claude-opus-4-7" = ["claude-sonnet-4-6"] }

Comportement : Lorsque Opus est rate-limited ou lent, utilisez automatiquement Sonnet (généralement 2 à 3 fois plus rapide et moins cher).

Configuration multi-régions

  • Pour les organisations avec des déploiements multi-régions :
# Région principale des États-Unis
default_provider = "anthropic"
default_model = "claude-sonnet-4-6"

[reliability]
# Utiliser le fournisseur de la région UE en cas de défaillance d'Anthropic aux États-Unis
fallback_providers = ["bedrock"]  # AWS Bedrock sur plusieurs régions
provider_retries = 3
provider_backoff_ms = 1000
  • Assurez-vous que chaque fournisseur de secours dispose d’informations d’identification dans votre environnement :
export ANTHROPIC_API_KEY="..."
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."

Comportement de rechargement à chaud

La section [reliability] est hot-reloadable. Pendant qu’un canal ou une passerelle est en cours d’exécution, les modifications apportées à config.toml prennent effet lors du prochain message entrant sans nécessiter de redémarrage.

Gestion des erreurs et déclencheurs de secours

Le comportement par défaut est déclenché par :

  • Timeout : Le fournisseur n’a pas répondu dans le délai de timeout configuré
  • Erreur de connexion: Échec réseau/DNS
  • Erreur d’authentification : Identifiants invalides (seuls les problèmes transitoires du service d’authentification entraîtent des nouvelles tentatives)
  • Limite de taux (429) : HTTP 429 ; entraîne d’abord la rotation de la clé API, puis le secours du fournisseur
  • Service indisponible (503) : problème temporaire de service
  • Modèle introuvable : Déclenche la chaîne de repli du modèle si configurée

Chute non déclenchée par :

  • Requête invalide (400) : Entrée malformée ; réessayer ne sera pas utile
  • Échec d’authentification permanent : Format de clé API invalide
  • **Erreurs de sortie du modèle** : Le modèle a répondu mais a retourné une erreur

Activité de secours de débogage

Activer les traces d’exécution pour déboguer le comportement de repli :

[observabilité]
runtime_trace_mode = "rolling"
runtime_trace_path = "state/runtime-trace.jsonl"

Puis interroge les traces :

- Afficher tous les événements de repli
zeroclaw doctor traces --contains "secours"

- Afficher uniquement les détails de retry du fournisseur.
zeroclaw doctor traces --contains "presta

**Afficher la rotation de la limite de débit**
zeroclaw doctor traces --contains "429"

Bonnes pratiques

  1. Par ordre de fiabilité : placez les fournisseurs les plus fiables en premier dans fallback_providers
  2. Vérifiez les chaînes de secours : Vérifiez le comportement de secours avant une utilisation en production
  3. Surveiller la rotation des clés API : Suivez les événements de limitation de débit pour savoir lorsque la rotation est active
  4. Conserver les modèles de repli sémantiquement semblables : Ne passez pas d’un modèle de raisonnement à un modèle de chat sans intention
  5. Utiliser les variables d’environnement : stocker les clés API sensibles dans les variables d’environnement, et non dans les fichiers de configuration.
  6. Documenter la logique de dépannage : Ajouter des commentaires dans le fichier de configuration pour expliquer la raison d’être de chaque mécanisme de dépannage.
  7. Vérifier les identifiants multi-modèles : Assurez-vous que tous les fournisseurs de repli ont des identifiants valides définis

Résolution des identifiants

Chaque fournisseur de repli résout les identifiants de manière indépendante en utilisant l’ordre de résolution standard :

  1. Identifiants explicites depuis la configuration/CLI
  2. Variable d’environnement spécifique au fournisseur
  3. Génération de secours : ZEROCLAW_API_KEY, puis API_KEY

La clé d’API du fournisseur principal n’est pas automatiquement réutilisée par les fournisseurs de repli. Définissez les identifiants pour chaque fournisseur séparément.

La traduction de la réponse est l’objet de la requête.

export OPENAI_API_KEY=“sk-...”
export ANTHROPIC_API_KEY="claude-..."
export GROQ_API_KEY="gsk-..."

Limites et contraintes

  • Maximum de fournisseurs de secours : limité par la taille du fichier de configuration (généralement plus de 100 chaînes sont supportées)
  • Chutes de secours du modèle maximal par modèle : Aucune limite stricte
  • Rotation de la clé API : Toutes les clés sont essayées avant le délai d’attente
  • Tentatives de nouvelle tentative : configurables par fournisseur avec une progression exponentielle
  • Budget total des délais d’attente : Cumulé sur les tentatives et les récupérations de secours ; le délai d’attente au niveau du canal s’applique toujours
